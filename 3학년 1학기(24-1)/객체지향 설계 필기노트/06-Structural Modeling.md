# CRC 카드 및 클래스 다이어그램 만들기(p.40)
## 객체 식별을 위한 4가지 접근 방식
- 구문 분석 / 브레인스토밍 / 공통 객체 목록 / 패턴

## 1. 구문 분석
- Use Case 설명서를 분석한다.
- 잠재적 객체, 속성, 함수 및 관계를 식별한다.
  \- 명사는 클래스이다.
  \- 동사는 함수이다.

## 구문분석 가이드라인
- 일반명사 또는 보통명사는 클래스의 객체를 의미한다.
- 고유명사 또는 직접 참조는 클래스의 인스턴스를 의미한다.
- 집합명사는 다른 클래스의 인스턴스 그룹으로 구성된 개체 클래스를 의미한다.
- 형용사는 객체의 속성을 의미한다.
- 동사는 함수를 의미한다.

---
# 1. 구문 분석(이어서)
- 존재 동사는 객체와 해당 클래스 간의 분류 관계를 의미한다.
- 소유동사는 집합 또는 연관 관계를 의미한다.(aggregation / association)
- 타동사는 함수를 의미한다.
- 자동사는 예외를 의미한다.
- 술어 또는 설명 동사구는 작업을 의미한다.
- 부사는 관계나 연산의 속성을 의미한다.

---
# 2. 공통 객체 목록
## : 시스템의 비즈니스 도메인에 공통된 객체 목록
## 객체 카테고리
- 물리적이고 유형의 것
  \- 책, 책상, 사무용품 등
- 사건(incident) : 비즈니스 영역에서 발생한 사건
  \- 회의, 비행, 공연, 사고 등
- 역할 : 사람들이 문제를 해결하는 역할
  \- 의사, 간호사, 환자 등
- 상호작용 : 비즈니스 도메인에서 발생하는 거래
  \- 판매 거래
- 장소, 조직, 정책 등과 같은 더 많은 것

---
# 3. 브레인스토밍
## : 아이디어를 제안하는 사람들
- 클래스(객체)의 초기 목록이 개발되었다.
- 다른 클래스에 대한 속성, 작업 및 관계는 두 번째 라운드에서 할당될 수 있다.

## 고려사항
- 진행자의 조율로 진행된다.
- 기능적 모델을 사용하지 않는다.(실제로 작동하는지의 여부는 생각하지 않는다.)
- 마음에서 나오는 잠재적인 대상의 집합이다.(생각에 제한을 두지 말아라.)

---
# 4. 패턴
## : 흔히 발생하는 문제에 대한 솔루션을 제공하는 유용한 협력 수업 그룹(특정한 문제의 해결책이 반복)

## 재사용 가능한 구성 요소

## 예 : 거래
- 거래 클래스
- 거래 라인 분류
- 아이템 클래스
- 위치 클래스
- 참가자 클래스

# 객체 ID 및 구조 모델링을 위한 단계
1. 사용 사례에 대한 텍스트 분석을 수행하여 CRC 카드를 만든다.
2. CRC 카드를 검토하여 추가 후보 클래스를 찾는다.
3. CRC 카드를 사용하여 각 사용 사례의 역할극을 수행한다.
4. CRC 카드를 기반으로 클래스 다이어그램을 만든다.
5. 누락되거나 불필요한 클래스, 속성, 작업 및 관계에 대한 구조 모델을 검토한다.
6. 유용한 패턴을 통합한다.
7. 구조 모델을 검토한다.
\* 해당 모델이 작고 잘 알려진 SW 모델이라면 CRC 카드 생성은 선택사항이다.

# 구조 모델링 단계
## 1. CRC 카드 생성
- Normal Flow, Sub Flow, Alternative Flow를 설명하는 Use Case 설명서에 대한 구문 분석을 수행한다.
- 때로는, 원래 요구 사항을 검토하여 다른 정보를 찾아본다.

## 2. CRC 카드 검토
- 추가 후보 클래스, 속성(attribute), 함수(operation) 및 관계가 누락되었는지 확인한다.
- 브레인스토밍 및 공통 개체 목록 접근 방식을 사용한다.

## 3. CRC 카드 롤플레잉
- 추가 개체, 속성, 관계 및 작업을 발견하는 데 도움이 되는 연습
- 팀 구성원은 이전에 식별된 행위자 및 개체와 관련된 역할을 수행한다.
- 팀 구성원은 Use Case 시나리오의 각 단계를 수행한다.
- 성공적인 결론에 도달할 때까지 문제를 발견하고 수정한다.

## 4. 클래스 다이어그램 만들기
- CRC 카드에 포함된 정보 사용
  \- 책임(Responsibilities) :: 클래스의 함수
  \- 속성(Attributes) :: 클래스의 속성
  \- 관계(Relationships) :: 일반화, 포함 및 연관 관계(generalization, aggregation, association)

## 5. 클래스 다이어그램 검토(Review)
- 누락되거나 불필요한 클래스, 속성, 함수 및 관계를 위해
- 악마의 대변자(devil's advocate)에 의해 - 일단 필요없어! 라고 생각해본다.

## 6. 패턴 통합
- 구조 모델을 발전시키기 위해
- 선택한 패턴과 클래스 모델 간의 클래스 비교
- 패턴을 모델에 통합하고 CRC 카드를 수정한다.

## 7. 모델 검토
- CRC 카드 및 클래스 모델
