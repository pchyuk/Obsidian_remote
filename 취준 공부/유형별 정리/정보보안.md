오늘 주제: 정보보안 - 접근통제 모델 DAC·MAC·RBAC

1. 핵심 요약
- 접근통제는 사용자가 어떤 자원에 접근할 수 있는지 허용하거나 제한하는 보안 기법입니다.
- DAC는 임의적 접근통제로, 자원 소유자가 접근 권한을 부여하거나 회수합니다.
- MAC은 강제적 접근통제로, 보안 등급과 정책에 따라 접근 여부가 결정됩니다.
- RBAC는 역할 기반 접근통제로, 사용자에게 직접 권한을 주기보다 역할에 권한을 부여합니다.
- 예를 들어 “인사팀 직원” 역할에 인사정보 조회 권한을 주고, 사용자를 그 역할에 배정하는 방식입니다.
- RBAC는 조직의 직무 구조와 잘 맞아 권한 관리가 쉽고, 인사 이동 시 역할만 바꾸면 됩니다.
- 시험에서는 “소유자 중심=DAC”, “보안 등급/정책 중심=MAC”, “직무·역할 중심=RBAC”로 구분하면 좋습니다.

2. 자주 나오는 포인트
- DAC: 자원 소유자가 권한을 결정, 유연하지만 권한 남용 가능성이 있음
- MAC: 보안 등급 기반, 군사·공공 보안 환경에서 자주 언급
- RBAC: 역할에 권한을 부여하고 사용자를 역할에 배정, 실무적 권한 관리에 적합

3. OX 문제
역할 기반 접근통제(RBAC)는 사용자에게 권한을 직접 하나씩 부여하기보다 역할에 권한을 부여하고, 사용자를 역할에 배정해 권한 관리를 단순화한다.

### 2026-04-30 18:00 - 정보보안 - SQL Injection과 방어 기법

[정보처리기사/전산직 필기]
오늘 주제: 정보보안 - SQL Injection과 방어 기법

1. 핵심 요약
- SQL Injection은 사용자의 입력값이 SQL문의 일부처럼 해석되도록 조작하는 공격입니다.
- 예를 들어 로그인 입력란에 조건을 항상 참으로 만드는 문자열을 넣어 인증을 우회하려는 방식이 대표적입니다.
- 동작 원리는 “입력값”과 “SQL 명령 구조”가 분리되지 않을 때 발생합니다.
- SQL문을 문자열로 직접 이어 붙이면 공격자가 입력한 값이 조건식, 주석, UNION 조회 등으로 해석될 수 있습니다.
- 방어의 핵심은 Prepared Statement, 즉 매개변수화 쿼리를 사용하는 것입니다.
- Prepared Statement는 SQL 구조를 먼저 고정하고, 사용자 입력은 나중에 값으로만 바인딩하므로 공격 문자열이 명령으로 실행되기 어렵습니다.
- 입력값 검증, DB 계정 최소 권한, 오류 메시지 노출 제한도 함께 중요합니다.
- SQLD 관점에서도 SQL 구조와 조건식 해석 방식을 이해하면 Injection 원리를 더 쉽게 잡을 수 있습니다.

2. 자주 나오는 포인트
- SQL Injection은 웹 보안 취약점이면서 DB 보안 위협으로 자주 출제됩니다.
- Prepared Statement / 바인드 변수는 입력값과 SQL 명령 구조를 분리하는 대표 방어 기법입니다.
- 단순히 특수문자만 차단하는 방식은 우회 가능성이 있어 충분한 방어로 보기 어렵습니다.

3. OX 문제
SQL Injection은 사용자가 입력한 값을 SQL문의 일부로 악의적으로 삽입해 인증 우회나 데이터 유출을 일으킬 수 있으므로, 입력값 검증과 Prepared Statement 같은 매개변수화 쿼리 사용이 방어에 도움이 된다.

[정보처리기사/전산직 필기]
오늘 주제: 정보보안 - 인증과 인가

1. 핵심 요약
- 인증(Authentication)은 사용자가 “누구인지” 확인하는 절차입니다.
- 대표적으로 아이디/비밀번호, OTP, 인증서, 생체인증 등이 있습니다.
- 인가(Authorization)는 인증된 사용자가 특정 자원에 접근할 “권한이 있는지” 판단하는 절차입니다.
- 예를 들어 로그인은 인증이고, 관리자 페이지 접근 가능 여부 확인은 인가입니다.
- 보안 흐름은 보통 인증 → 인가 순서로 진행됩니다.
- 인증에 성공해도 모든 자원에 접근할 수 있는 것은 아닙니다.
- 접근통제에서는 사용자, 역할, 권한을 기준으로 인가 여부를 판단합니다.
- 시험에서는 인증과 인가의 개념을 서로 바꿔 묻는 문제가 자주 나옵니다.

2. 자주 나오는 포인트
- 인증: 신원 확인, “너 누구야?”
- 인가: 권한 확인, “너 이거 해도 돼?”
- 인증 성공 ≠ 모든 권한 보유

3. OX 문제
인증은 사용자가 누구인지 확인하는 절차이고, 인가는 인증된 사용자가 특정 자원에 접근할 권한이 있는지 판단하는 절차이다.

1. 핵심 요약
- 디지털 서명은 전자문서나 메시지가 “누가 보냈고, 중간에 바뀌지 않았는지” 확인하는 기술입니다.
- 보통 원문 전체를 개인키로 암호화하는 것이 아니라, 원문의 해시값을 만든 뒤 그 해시값에 송신자의 개인키로 서명합니다.
- 수신자는 송신자의 공개키로 서명을 검증하고, 자신이 계산한 해시값과 비교합니다.

- 또한 나중에 “내가 보낸 것이 아니다”라고 부인하기 어렵게 하므로 부인 방지에 활용됩니다.
- 단, 디지털 서명 자체의 주목적은 기밀성보다 인증·무결성·부인 방지입니다.
- 기밀성을 원하면 보통 수신자의 공개키로 암호화해 수신자의 개인키로만 복호화하게 합니다.

2. 자주 나오는 포인트
- 디지털 서명: 송신자 개인키로 서명, 송신자 공개키로 검증
- 제공 기능: 송신자 인증, 무결성, 부인 방지
- 기밀성 암호화와 구분: 기밀성은 수신자 공개키로 암호화, 수신자 개인키로 복호화

[정보처리기사/전산직 필기]
오늘 주제: 정보보안 - XSS 공격과 방어

1. 핵심 요약
- XSS는 Cross-Site Scripting의 약자로, 웹 페이지에 악성 스크립트를 삽입하는 공격입니다.
- 공격자가 게시글, 댓글, 검색어, URL 파라미터 등에 스크립트를 넣으면 피해자의 브라우저에서 실행될 수 있습니다.
- 핵심은 서버가 받은 입력값을 검증하지 않거나, 출력할 때 안전하게 처리하지 않아 브라우저가 이를 코드로 해석하는 것입니다.
- 피해자는 정상 사이트에 접속했다고 생각하지만, 브라우저 안에서는 공격자의 스크립트가 실행될 수 있습니다.
- 이로 인해 쿠키 탈취, 세션 하이재킹, 피싱 화면 표시, 악성 요청 전송 등이 발생할 수 있습니다.
- SQL Injection이 DB 질의를 조작하는 공격이라면, XSS는 사용자의 브라우저에서 스크립트가 실행되는 공격입니다.
- 방어는 입력값 검증, 출력 인코딩, 쿠키 HttpOnly 설정, CSP 적용 등이 중요합니다.
- 시험에서는 “스크립트 삽입”, “브라우저 실행”, “출력 인코딩”을 키워드로 잡으면 좋습니다.

2. 자주 나오는 포인트
- XSS: 웹 페이지에 악성 스크립트를 삽입해 사용자 브라우저에서 실행.
- SQL Injection과 구분: SQL Injection은 DB 명령 조작, XSS는 클라이언트 스크립트 실행.
- 방어: 입력 검증만으로 부족할 수 있고, 출력 시 문맥에 맞는 인코딩이 중요.

3. OX 문제
XSS는 웹 페이지에 악성 스크립트를 삽입해 다른 사용자의 브라우저에서 실행되도록 하는 공격이며, 입력값 검증과 출력 인코딩이 방어에 도움이 된다.
