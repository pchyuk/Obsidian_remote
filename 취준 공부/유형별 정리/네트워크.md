### 2026-04-26 19:30 - 네트워크 - TCP와 UDP

지난 문제 정답

정답: O

해설: 제3정규형(3NF)은 2NF를 만족하면서 이행적 함수 종속을 제거하는 정규형입니다.  
예를 들어 `학번 → 학과번호`, `학과번호 → 학과명`이면 `학번 → 학과명`은 이행적 종속입니다.  
이런 경우 학과 정보를 별도 테이블로 분리해 중복과 이상 현상을 줄입니다.

오늘의 주제: 네트워크 - TCP와 UDP

TCP와 UDP는 전송 계층에서 사용하는 대표적인 프로토콜입니다.  
TCP는 연결형 프로토콜로, 데이터를 보내기 전에 연결 설정 과정을 거칩니다.  
대표적으로 3-way handshake를 통해 송신자와 수신자가 통신 준비 상태인지 확인합니다.  
TCP는 데이터의 순서 보장, 오류 검출, 재전송, 흐름 제어, 혼잡 제어를 제공해 신뢰성이 높습니다.  
반면 UDP는 비연결형 프로토콜로, 연결 설정 없이 데이터그램을 바로 전송합니다.  
그래서 TCP보다 오버헤드가 작고 빠르지만, 데이터 손실이나 순서 뒤바뀜을 자체적으로 보장하지 않습니다.  
UDP는 DNS, VoIP, 실시간 스트리밍처럼 속도와 지연 시간이 중요한 서비스에서 자주 사용됩니다.

자주 나오는 포인트

- TCP: 연결형, 신뢰성 높음, 순서 보장, 재전송 가능
- UDP: 비연결형, 빠름, 오버헤드 작음, 신뢰성 보장 약함
- 3-way handshake는 TCP 연결 설정 과정

OX 문제

UDP는 연결 설정 과정이 없고, 데이터그램 단위로 전송하며, TCP보다 전송 신뢰성을 항상 더 높게 보장한다.

오늘 주제: 네트워크 - DNS와 이름 해석

1. 핵심 요약
- DNS는 Domain Name System의 약자로, 사람이 기억하기 쉬운 도메인 이름을 실제 통신에 필요한 IP 주소로 바꿔 주는 시스템입니다.
- 예를 들어 `www.example.com`에 접속하면 컴퓨터는 결국 해당 서버의 IP 주소를 알아야 패킷을 보낼 수 있습니다.
- DNS는 하나의 중앙 서버가 전부 처리하는 방식이 아니라, 루트 DNS → TLD DNS → 권한 있는 DNS 서버처럼 계층적으로 동작합니다.
- 루트 DNS는 `.com`, `.kr` 같은 최상위 도메인 서버 위치를 알려 주고, TLD DNS는 해당 도메인을 관리하는 서버를 안내합니다.
- 권한 있는 DNS 서버는 최종적으로 특정 도메인에 대응하는 IP 주소 같은 레코드를 제공합니다.
- 자주 쓰는 DNS 결과는 캐시에 저장됩니다. 그래서 같은 사이트에 다시 접속할 때 매번 전체 조회 과정을 반복하지 않아도 됩니다.
- DNS 레코드에는 IPv4 주소를 나타내는 A 레코드, IPv6 주소를 나타내는 AAAA 레코드, 별칭을 나타내는 CNAME, 메일 서버를 나타내는 MX 등이 있습니다.
- 시험에서는 “도메인 이름 ↔ IP 주소 변환”, “계층적 분산 구조”, “캐싱”, “주요 레코드 종류”를 함께 묶어 기억하면 좋습니다.

2. 자주 나오는 포인트
- DNS는 도메인 이름을 IP 주소로 변환하는 이름 해석 시스템입니다.
- 루트 DNS, TLD DNS, 권한 있는 DNS 서버로 이어지는 계층 구조를 사용합니다.
- 캐싱은 DNS 조회 속도를 높이고 네트워크 부하를 줄이지만, TTL 동안 오래된 정보가 남을 수도 있습니다.

3. OX 문제
DNS는 사람이 읽기 쉬운 도메인 이름을 IP 주소로 변환해 주는 시스템이며, 일반적으로 계층적 구조와 캐싱을 이용해 조회 효율을 높인다.

오늘 주제: 네트워크 - IP 주소와 서브넷 마스크

1. 핵심 요약
- IP 주소는 네트워크에서 장비를 식별하기 위한 논리적 주소입니다.
- IPv4는 보통 `192.168.0.10`처럼 10진수 4개로 표현하지만, 내부적으로는 32비트입니다.
- 서브넷 마스크는 IP 주소 중 어디까지가 “네트워크 부분”이고 어디부터가 “호스트 부분”인지 구분하는 기준입니다.
- 예를 들어 `/24`는 앞 24비트가 네트워크 부분이고, 나머지 8비트가 호스트 부분이라는 뜻입니다.
- 같은 네트워크 주소를 가진 장비들은 같은 서브넷에 있다고 봅니다.
- 같은 서브넷 안에서는 보통 라우터를 거치지 않고 직접 통신할 수 있습니다.
- 다른 서브넷으로 가려면 기본 게이트웨이, 즉 라우터를 통해 패킷이 전달됩니다.
- 그래서 IP 주소와 서브넷 마스크를 함께 봐야 “같은 네트워크인지, 다른 네트워크인지” 판단할 수 있습니다.
- 시험에서는 `/24`, `/26` 같은 CIDR 표기와 사용 가능한 호스트 수 계산이 자주 나옵니다.

2. 자주 나오는 포인트
- 서브넷 마스크: 네트워크 부분과 호스트 부분을 구분하는 값
- 같은 서브넷 여부: IP 주소에 서브넷 마스크를 적용한 네트워크 주소가 같은지 확인
- 호스트 수 계산: 호스트 비트가 n개면 사용 가능 호스트 수는 보통 `2ⁿ - 2`

3. OX 문제
서브넷 마스크는 IP 주소에서 네트워크 부분과 호스트 부분을 구분하는 기준이며, 같은 서브넷에 속한 장비들은 일반적으로 라우터를 거치지 않고 직접 통신할 수 있다.

오늘 주제: 네트워크 - HTTP와 무상태성

1. 핵심 요약
- HTTP는 웹 브라우저 같은 클라이언트와 웹 서버가 데이터를 주고받기 위한 응용 계층 프로토콜입니다.
- 기본 구조는 클라이언트가 요청(Request)을 보내고, 서버가 응답(Response)을 돌려주는 방식입니다.
- 예를 들어 브라우저가 “이 페이지 보여줘”라고 요청하면 서버는 HTML, 이미지, JSON 같은 데이터를 응답합니다.
- HTTP의 중요한 특징은 무상태성(stateless)입니다. 즉, 서버는 기본적으로 이전 요청을 자동으로 기억하지 않습니다.
- 그래서 로그인 유지, 장바구니, 사용자 설정처럼 “이전 상태”가 필요한 기능은 쿠키, 세션, 토큰 등을 함께 사용합니다.
- HTTP는 평문 통신이고, HTTPS는 HTTP에 SSL/TLS 암호화를 적용해 기밀성과 무결성을 강화한 방식입니다.
- 동작 원리는 보통 TCP 연결 위에서 요청 메시지와 응답 메시지를 주고받는 형태로 이해하면 됩니다.
- 시험에서는 HTTP가 어느 계층인지, 요청/응답 구조, 무상태성, HTTPS와의 차이가 자주 나옵니다.

2. 자주 나오는 포인트
- HTTP: 응용 계층 프로토콜, 요청/응답 방식
- 무상태성: 이전 요청 정보를 기본적으로 기억하지 않음
- HTTPS: HTTP + SSL/TLS, 암호화로 보안 강화

3. OX 문제
HTTP는 웹 클라이언트와 서버가 요청과 응답을 주고받기 위한 응용 계층 프로토콜이며, 기본적으로 각 요청은 독립적으로 처리되므로 상태 유지를 위해 쿠키나 세션 같은 보조 수단을 사용할 수 있다.

오늘 주제: 네트워크 - TCP 흐름 제어와 혼잡 제어

1. 핵심 요약

- 쉽게 말해 “받는 쪽 버퍼가 넘치지 않게 하는 제어”입니다.

- 대표 개념으로 혼잡 윈도우(cwnd), 슬로 스타트, 혼잡 회피, 빠른 재전송이 자주 나옵니다.
- 정리하면 흐름 제어는 수신자 보호, 혼잡 제어는 네트워크 보호입니다.

2. 자주 나오는 포인트
- 흐름 제어: 수신 버퍼 초과 방지, 수신 윈도우(rwnd), 송신자-수신자 간 속도 조절
- 혼잡 제어: 네트워크 과부하 방지, 혼잡 윈도우(cwnd), 슬로 스타트/혼잡 회피
- 시험에서는 “흐름 제어 = 수신자 기준”, “혼잡 제어 = 네트워크 상태 기준” 구분이 핵심입니다.

3. OX 문제
TCP의 흐름 제어는 수신자의 처리 능력을 넘지 않도록 송신량을 조절하는 것이고, 혼잡 제어는 네트워크 내부의 혼잡을 줄이기 위해 전송 속도를 조절하는 것이다.

오늘 주제: 네트워크 - OSI 7계층과 TCP/IP 모델

1. 핵심 요약
- OSI 7계층은 네트워크 통신 기능을 역할별로 나눈 참조 모델입니다.
- 아래 계층은 실제 전송에 가깝고, 위 계층은 사용자 서비스에 가깝습니다.
- 1계층 물리 계층은 전기 신호, 케이블, 허브처럼 비트 전송을 다룹니다.

해설: WITH CHECK OPTION은 뷰를 통해 INSERT나 UPDATE를 할 때, 변경 후의 행이 뷰의 정의 조건을 계속 만족하는지 검사하는 옵션입니다. 예를 들어 “부서번호=10인 사원만 보이는 뷰”를 통해 부서번호를 20으로 바꾸면 뷰 조건을 벗어나므로 제한될 수 있습니다. 뷰는 조회 편의성과 보안에 유용하지만, 모든 뷰가 자유롭게 갱신 가능한 것은 아니라는 점도 함께 기억하세요.

오늘 주제: 네트워크 - 오류 제어와 ARQ

1. 핵심 요약
- 오류 제어는 데이터 전송 중 비트 오류, 프레임 손상, 손실 등이 생겼을 때 이를 검출하고 복구하는 기능입니다.
- ARQ는 Automatic Repeat reQuest의 약자로, 오류가 있거나 ACK가 오지 않으면 데이터를 다시 보내는 재전송 기반 오류 제어 방식입니다.
- 기본 흐름은 송신자가 데이터를 보내고, 수신자가 정상 수신 시 ACK를 보내며, 오류가 있으면 NAK를 보내거나 ACK가 오지 않아 타임아웃이 발생합니다.
- Stop-and-Wait ARQ는 하나 보내고 ACK를 기다리는 방식이라 단순하지만 회선 이용 효율이 낮습니다.
- Go-Back-N ARQ는 여러 프레임을 연속 전송하다가 오류가 나면 오류 지점 이후 프레임들을 다시 보냅니다.
- Selective Repeat ARQ는 오류가 난 프레임만 선택적으로 재전송해 효율이 좋지만, 수신 버퍼와 순서 관리가 더 복잡합니다.
- 핵심은 ARQ가 “흐름 제어”가 아니라 “오류 제어”에 가깝고, 신뢰성을 높이기 위해 ACK/NAK/타임아웃/재전송을 사용한다는 점입니다.
- TCP의 신뢰성 보장도 ACK, 순서 번호, 재전송 같은 개념과 연결해서 이해하면 좋습니다.

2. 자주 나오는 포인트
- ARQ = 오류 검출 후 재전송으로 신뢰성 확보.
- Stop-and-Wait은 단순하지만 비효율, Selective Repeat은 효율적이지만 구현 복잡.
- 흐름 제어는 수신자 처리 능력에 맞춰 속도 조절, 오류 제어는 손상·손실 데이터 복구가 핵심입니다.
