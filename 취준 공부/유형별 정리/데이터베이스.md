### 2026-04-27 13:00 - SQL/DB - INNER JOIN과 OUTER JOIN

[정보처리기사/전산직 필기]
오늘 주제: SQL/DB - INNER JOIN과 OUTER JOIN

1. 핵심 요약
- JOIN은 두 개 이상의 테이블을 공통 컬럼이나 조건을 기준으로 연결해 하나의 결과처럼 조회하는 방법입니다.
- INNER JOIN은 조인 조건을 만족하는 행만 가져옵니다. 쉽게 말해 두 테이블의 “매칭되는 데이터”만 결과에 남습니다.
- OUTER JOIN은 조건에 맞지 않는 행도 보존할 수 있는 조인입니다.
- LEFT OUTER JOIN은 왼쪽 테이블의 행을 모두 남기고, 오른쪽 테이블에 매칭되는 값이 없으면 NULL로 채웁니다.
- RIGHT OUTER JOIN은 오른쪽 테이블을 기준으로 모두 남기고, FULL OUTER JOIN은 양쪽 모두의 미매칭 행까지 남깁니다.
- SQLD에서도 JOIN은 SELECT 문 핵심 파트로 자주 나옵니다. 특히 INNER JOIN과 OUTER JOIN의 결과 행 차이를 묻는 문제가 많습니다.
- JOIN 조건을 빠뜨리면 의도치 않은 카티션 곱이 발생해 행 수가 폭증할 수 있습니다.
- 시험에서는 “매칭되는 것만?”인지 “한쪽 기준으로 다 남기는지?”를 먼저 판단하면 헷갈림이 줄어듭니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: INNER JOIN은 조건을 만족하는 행만 출력합니다.
- SQLD 포인트: LEFT OUTER JOIN에서 오른쪽 테이블 값이 없으면 해당 컬럼은 NULL로 표시됩니다.
- 실무/시험 공통: JOIN 조건 누락 시 불필요한 조합 결과가 생길 수 있습니다.

3. OX 문제
INNER JOIN은 두 테이블에서 조인 조건을 만족하는 행만 결과에 포함하며, 조건에 맞지 않는 한쪽 테이블의 행은 결과에서 제외된다.

- 예를 들어 계좌 이체는 “A 계좌 출금”과 “B 계좌 입금”이 함께 성공해야 하나의 트랜잭션입니다.
- 트랜잭션의 대표 성질은 ACID입니다: 원자성, 일관성, 고립성, 지속성.
- 원자성은 트랜잭션 내 작업이 전부 성공하거나 전부 실패해야 한다는 성질입니다.
- 중간에 오류가 나면 일부 결과만 남기지 않고 ROLLBACK으로 이전 상태로 되돌려야 합니다.
- 반대로 모든 작업이 정상 완료되면 COMMIT을 통해 변경 내용을 확정합니다.
- SQLD에서도 트랜잭션, COMMIT, ROLLBACK, TCL 개념은 자주 나옵니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: COMMIT은 변경 확정, ROLLBACK은 변경 취소입니다.
- 원자성은 “All or Nothing”으로 기억하면 좋습니다.
- 트랜잭션 제어어 TCL에는 COMMIT, ROLLBACK, SAVEPOINT가 포함됩니다.

3. OX 문제
트랜잭션의 원자성은 트랜잭션 내 연산이 모두 수행되거나 모두 수행되지 않아야 함을 의미하며, 중간에 오류가 발생하면 이미 수행된 변경도 되돌릴 수 있어야 한다.

1. 핵심 요약
- 인덱스는 테이블에서 원하는 데이터를 빠르게 찾기 위한 자료구조입니다.
- 책 뒤의 “찾아보기”처럼, 특정 컬럼 값을 기준으로 데이터 위치를 빠르게 찾게 도와줍니다.
- 인덱스가 없으면 DBMS가 테이블 전체를 처음부터 끝까지 확인하는 Full Table Scan을 할 수 있습니다.
- 인덱스가 잘 잡혀 있으면 WHERE 조건, JOIN 조건, ORDER BY, GROUP BY 처리에서 검색 범위를 줄일 수 있습니다.
- 보통 B-Tree 인덱스가 자주 쓰이며, 범위 검색과 정렬된 탐색에 유리합니다.
- 하지만 인덱스는 별도 저장 공간을 차지하고, 데이터가 변경될 때 인덱스도 함께 갱신해야 합니다.
- 그래서 INSERT, UPDATE, DELETE가 많은 테이블에 인덱스를 너무 많이 만들면 오히려 성능이 떨어질 수 있습니다.
- SQLD에서도 자주 보는 포인트는 “인덱스는 조회 성능을 높일 수 있지만 무조건 많을수록 좋은 것은 아니다”입니다.
- 선택도가 높은 컬럼, 즉 값의 종류가 다양해서 조건으로 걸었을 때 결과 행을 많이 줄일 수 있는 컬럼에 인덱스를 두는 것이 유리합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 인덱스는 검색 속도 향상 목적이지만 저장 공간과 갱신 비용이 발생합니다.
- SQLD 포인트: WHERE, JOIN, ORDER BY에 자주 쓰이는 컬럼이 인덱스 후보가 될 수 있습니다.
- 시험 함정: 인덱스가 많으면 SELECT는 빨라질 수 있지만, DML 성능은 나빠질 수 있습니다.

인덱스는 검색 조건에 맞는 행을 더 빠르게 찾도록 도와줄 수 있지만, INSERT·UPDATE·DELETE가 자주 발생하는 테이블에서는 인덱스 유지 비용 때문에 성능 부담이 생길 수 있다.

### 2026-04-27 20:00 - SQL/DB - GROUP BY와 HAVING

[정보처리기사/전산직 필기]
오늘 주제: SQL/DB - GROUP BY와 HAVING

1. 핵심 요약
- GROUP BY는 여러 행을 특정 컬럼 기준으로 묶어 “그룹 단위”로 집계할 때 사용합니다.
- 예를 들어 부서별 평균 급여, 상품별 판매 수량, 회원별 주문 횟수처럼 “~별”이라는 말이 나오면 GROUP BY를 떠올리면 됩니다.
- COUNT, SUM, AVG, MAX, MIN 같은 집계 함수는 그룹별 결과를 계산할 때 자주 함께 쓰입니다.
- WHERE는 GROUP BY보다 먼저 실행되어, 그룹을 만들기 전 개별 행을 필터링합니다.
- HAVING은 GROUP BY 이후에 만들어진 그룹 결과를 조건으로 걸러낼 때 사용합니다.
- 그래서 “급여가 3000 이상인 직원만 모아서 부서별 평균”은 WHERE, “평균 급여가 5000 이상인 부서만 출력”은 HAVING입니다.
- SQLD에서도 SELECT 처리 순서와 WHERE/HAVING 구분은 자주 나오는 포인트입니다.
- 실무식으로 이해하면 WHERE는 재료 고르기, GROUP BY는 재료 묶기, HAVING은 완성된 묶음 중 조건에 맞는 것만 고르기입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 집계 함수 조건은 보통 HAVING에 작성합니다.
- SQLD 포인트: SELECT절에 집계 함수가 아닌 컬럼이 나오면 보통 GROUP BY에 포함되어야 합니다.
- 실행 흐름 감각: WHERE → GROUP BY → HAVING → SELECT → ORDER BY 순서로 이해하면 문제 풀이가 쉬워집니다.

3. OX 문제
WHERE절은 그룹화되기 전 개별 행을 먼저 걸러내고, HAVING절은 GROUP BY로 묶인 그룹에 대한 조건을 걸 때 사용한다.

- 정처기/SQLD 공통: SELECT는 DML, CREATE/ALTER/DROP은 DDL, GRANT/REVOKE는 DCL, COMMIT/ROLLBACK은 TCL.

- 대표 명령어는 GRANT와 REVOKE입니다.
- GRANT는 사용자나 역할에게 SELECT, INSERT, UPDATE, DELETE 같은 권한을 부여합니다.
- REVOKE는 이미 부여된 권한을 회수합니다.
- 예를 들어 특정 사용자에게 사원 테이블 조회 권한만 주면, 그 사용자는 데이터를 볼 수는 있지만 수정은 못 하게 제한할 수 있습니다.
- SQLD에서도 DML·DDL·DCL·TCL 분류는 자주 나옵니다.
- DML은 데이터 조작, DDL은 객체 정의, DCL은 권한 제어, TCL은 트랜잭션 제어로 구분하면 됩니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: GRANT는 권한 부여, REVOKE는 권한 회수입니다.
- DCL은 테이블 생성·삭제가 아니라 접근 권한 관리와 관련됩니다.
- DML: SELECT/INSERT/UPDATE/DELETE, DDL: CREATE/ALTER/DROP, TCL: COMMIT/ROLLBACK과 구분하세요.

3. OX 문제
DCL은 데이터베이스 객체에 대한 권한을 부여하거나 회수하는 명령어로, 대표적으로 GRANT와 REVOKE가 있다.

### 2026-05-01 17:00 - SQL/DB - DDL과 DELETE·TRUNCATE·DROP 구분

[정보처리기사/전산직 필기]
오늘 주제: SQL/DB - DDL과 DELETE·TRUNCATE·DROP 구분

1. 핵심 요약
- DDL은 Data Definition Language, 즉 데이터베이스 구조를 정의·변경·삭제하는 명령어입니다.
- 대표 명령어는 CREATE, ALTER, DROP, TRUNCATE입니다.
- CREATE는 테이블·뷰·인덱스 같은 객체를 만들고, ALTER는 이미 만든 객체의 구조를 변경합니다.
- DROP은 테이블 같은 객체 자체를 삭제합니다. 구조와 데이터가 함께 사라진다고 이해하면 됩니다.
- DELETE는 헷갈리기 쉽지만 DDL이 아니라 DML입니다. 테이블 구조는 유지하고 행 데이터만 삭제합니다.
- TRUNCATE는 테이블 구조는 남기고 전체 데이터를 빠르게 삭제하는 DDL 성격의 명령으로 자주 출제됩니다.
- SQLD에서도 DELETE·TRUNCATE·DROP 구분은 단골입니다. “행 삭제냐, 전체 데이터 초기화냐, 객체 삭제냐”로 나누면 좋습니다.
- 실무 감각으로 보면 DELETE는 조건 삭제, TRUNCATE는 통째 비우기, DROP은 테이블 자체 제거입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: DDL은 CREATE, ALTER, DROP, TRUNCATE / DML은 SELECT, INSERT, UPDATE, DELETE.
- DELETE는 WHERE절로 일부 행 삭제 가능, DROP은 객체 자체 삭제라 테이블 구조도 사라짐.
- TRUNCATE는 전체 행 삭제에 가깝지만 DDL로 분류되는 경우가 많아 DELETE와 구분해야 함.

3. OX 문제
DDL의 DROP은 테이블 구조 자체를 제거하는 명령이고, DELETE는 테이블 구조는 유지한 채 행 데이터를 삭제하는 DML 명령이다.

### 2026-05-01 20:00 - SQL/DB - WHERE 조건과 LIKE 패턴 검색

[정보처리기사/전산직 필기]
오늘 주제: SQL/DB - WHERE 조건과 LIKE 패턴 검색

1. 핵심 요약
- WHERE절은 테이블에서 조건에 맞는 행만 걸러낼 때 사용합니다.
- SELECT가 “어떤 컬럼을 볼지” 정한다면, WHERE는 “어떤 행을 볼지” 정하는 역할입니다.
- LIKE는 문자열이 특정 패턴과 일치하는지 확인할 때 쓰는 조건 연산자입니다.
- `%`는 0개 이상의 임의 문자를 의미합니다. 예를 들어 `LIKE '김%'`는 김으로 시작하는 값을 찾습니다.
- `_`는 정확히 1개의 임의 문자를 의미합니다. 예를 들어 `LIKE 'A_'`는 A로 시작하는 두 글자 값을 찾습니다.
- `BETWEEN A AND B`는 범위 조건, `IN (...)`은 여러 후보 값 중 포함 여부를 검사합니다.
- NULL은 일반 비교 연산자인 `= NULL`로 찾지 않고, 반드시 `IS NULL` 또는 `IS NOT NULL`을 사용합니다.
- SQLD에서도 WHERE, LIKE, IN, BETWEEN, IS NULL은 SELECT 기본 문법에서 자주 나오는 포인트입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: `LIKE 'A%'`는 A로 시작, `LIKE '%A'`는 A로 끝, `LIKE '%A%'`는 A 포함입니다.
- SQLD 포인트: `%`는 0개 이상, `_`는 정확히 1개 문자라는 차이를 구분해야 합니다.
- 주의 포인트: NULL 비교는 `= NULL`이 아니라 `IS NULL`입니다.

3. OX 문제
SQL의 WHERE절에서 LIKE는 문자열 패턴 검색에 사용되며, 와일드카드 %는 0개 이상의 임의 문자, _는 정확히 1개의 임의 문자를 의미한다.

### 2026-05-02 09:02 - SQL/DB - 데이터 모델링 식별자와 주식별자

오늘 주제: SQL/DB - 데이터 모델링 식별자와 주식별자

1. 핵심 요약
- 식별자는 엔터티 안의 여러 인스턴스를 서로 구분해 주는 속성 또는 속성 집합입니다.
- 예를 들어 학생 엔터티에서 학번이 각 학생을 유일하게 구분한다면 학번은 식별자가 될 수 있습니다.
- 후보 식별자는 유일성과 최소성을 만족해 식별자로 쓸 수 있는 후보들입니다.
- 그중 업무적으로 대표로 선택한 식별자를 주식별자라고 합니다.
- 관계형 모델로 바꾸면 주식별자는 보통 기본키(PK)가 됩니다.
- 선택되지 않은 후보 식별자는 대체키가 될 수 있습니다.
- 식별자는 가능하면 값이 자주 바뀌지 않아야 하고, NULL이 되면 안 됩니다.
- SQLD에서도 데이터 모델링 파트에서 엔터티, 속성, 관계와 함께 식별자 개념을 자주 묻습니다.
- 정처기에서는 기본키, 후보키, 대체키, 외래키와 연결해서 출제되는 경우가 많습니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 주식별자 = 후보 식별자 중 대표로 선택한 식별자
- SQLD 포인트: 식별자는 유일성, 최소성, 불변성, 존재성을 만족하는지 확인
- DB 포인트: 주식별자는 테이블 변환 시 기본키가 되고, 관계 설정 시 외래키와 연결될 수 있음

3. OX 문제
데이터 모델링에서 식별자는 엔터티의 인스턴스를 유일하게 구분하는 속성 또는 속성 집합이며, 후보 식별자 중 대표로 선택된 것이 주식별자이다.
