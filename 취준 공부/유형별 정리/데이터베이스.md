### 2026-04-27 13:00 - SQL/DB - INNER JOIN과 OUTER JOIN

[정보처리기사/전산직 필기]
오늘 주제: SQL/DB - INNER JOIN과 OUTER JOIN

1. 핵심 요약
- JOIN은 두 개 이상의 테이블을 공통 컬럼이나 조건을 기준으로 연결해 하나의 결과처럼 조회하는 방법입니다.
- INNER JOIN은 조인 조건을 만족하는 행만 가져옵니다. 쉽게 말해 두 테이블의 “매칭되는 데이터”만 결과에 남습니다.
- OUTER JOIN은 조건에 맞지 않는 행도 보존할 수 있는 조인입니다.
- LEFT OUTER JOIN은 왼쪽 테이블의 행을 모두 남기고, 오른쪽 테이블에 매칭되는 값이 없으면 NULL로 채웁니다.
- RIGHT OUTER JOIN은 오른쪽 테이블을 기준으로 모두 남기고, FULL OUTER JOIN은 양쪽 모두의 미매칭 행까지 남깁니다.
- SQLD에서도 JOIN은 SELECT 문 핵심 파트로 자주 나옵니다. 특히 INNER JOIN과 OUTER JOIN의 결과 행 차이를 묻는 문제가 많습니다.
- JOIN 조건을 빠뜨리면 의도치 않은 카티션 곱이 발생해 행 수가 폭증할 수 있습니다.
- 시험에서는 “매칭되는 것만?”인지 “한쪽 기준으로 다 남기는지?”를 먼저 판단하면 헷갈림이 줄어듭니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: INNER JOIN은 조건을 만족하는 행만 출력합니다.
- SQLD 포인트: LEFT OUTER JOIN에서 오른쪽 테이블 값이 없으면 해당 컬럼은 NULL로 표시됩니다.
- 실무/시험 공통: JOIN 조건 누락 시 불필요한 조합 결과가 생길 수 있습니다.

3. OX 문제
INNER JOIN은 두 테이블에서 조인 조건을 만족하는 행만 결과에 포함하며, 조건에 맞지 않는 한쪽 테이블의 행은 결과에서 제외된다.

- 예를 들어 계좌 이체는 “A 계좌 출금”과 “B 계좌 입금”이 함께 성공해야 하나의 트랜잭션입니다.
- 트랜잭션의 대표 성질은 ACID입니다: 원자성, 일관성, 고립성, 지속성.
- 원자성은 트랜잭션 내 작업이 전부 성공하거나 전부 실패해야 한다는 성질입니다.
- 중간에 오류가 나면 일부 결과만 남기지 않고 ROLLBACK으로 이전 상태로 되돌려야 합니다.
- 반대로 모든 작업이 정상 완료되면 COMMIT을 통해 변경 내용을 확정합니다.
- SQLD에서도 트랜잭션, COMMIT, ROLLBACK, TCL 개념은 자주 나옵니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: COMMIT은 변경 확정, ROLLBACK은 변경 취소입니다.
- 원자성은 “All or Nothing”으로 기억하면 좋습니다.
- 트랜잭션 제어어 TCL에는 COMMIT, ROLLBACK, SAVEPOINT가 포함됩니다.

3. OX 문제
트랜잭션의 원자성은 트랜잭션 내 연산이 모두 수행되거나 모두 수행되지 않아야 함을 의미하며, 중간에 오류가 발생하면 이미 수행된 변경도 되돌릴 수 있어야 한다.

1. 핵심 요약
- 인덱스는 테이블에서 원하는 데이터를 빠르게 찾기 위한 자료구조입니다.
- 책 뒤의 “찾아보기”처럼, 특정 컬럼 값을 기준으로 데이터 위치를 빠르게 찾게 도와줍니다.
- 인덱스가 없으면 DBMS가 테이블 전체를 처음부터 끝까지 확인하는 Full Table Scan을 할 수 있습니다.
- 인덱스가 잘 잡혀 있으면 WHERE 조건, JOIN 조건, ORDER BY, GROUP BY 처리에서 검색 범위를 줄일 수 있습니다.
- 보통 B-Tree 인덱스가 자주 쓰이며, 범위 검색과 정렬된 탐색에 유리합니다.
- 하지만 인덱스는 별도 저장 공간을 차지하고, 데이터가 변경될 때 인덱스도 함께 갱신해야 합니다.
- 그래서 INSERT, UPDATE, DELETE가 많은 테이블에 인덱스를 너무 많이 만들면 오히려 성능이 떨어질 수 있습니다.
- SQLD에서도 자주 보는 포인트는 “인덱스는 조회 성능을 높일 수 있지만 무조건 많을수록 좋은 것은 아니다”입니다.
- 선택도가 높은 컬럼, 즉 값의 종류가 다양해서 조건으로 걸었을 때 결과 행을 많이 줄일 수 있는 컬럼에 인덱스를 두는 것이 유리합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 인덱스는 검색 속도 향상 목적이지만 저장 공간과 갱신 비용이 발생합니다.
- SQLD 포인트: WHERE, JOIN, ORDER BY에 자주 쓰이는 컬럼이 인덱스 후보가 될 수 있습니다.
- 시험 함정: 인덱스가 많으면 SELECT는 빨라질 수 있지만, DML 성능은 나빠질 수 있습니다.

인덱스는 검색 조건에 맞는 행을 더 빠르게 찾도록 도와줄 수 있지만, INSERT·UPDATE·DELETE가 자주 발생하는 테이블에서는 인덱스 유지 비용 때문에 성능 부담이 생길 수 있다.

### 2026-04-27 20:00 - SQL/DB - GROUP BY와 HAVING

[정보처리기사/전산직 필기]
오늘 주제: SQL/DB - GROUP BY와 HAVING

1. 핵심 요약
- GROUP BY는 여러 행을 특정 컬럼 기준으로 묶어 “그룹 단위”로 집계할 때 사용합니다.
- 예를 들어 부서별 평균 급여, 상품별 판매 수량, 회원별 주문 횟수처럼 “~별”이라는 말이 나오면 GROUP BY를 떠올리면 됩니다.
- COUNT, SUM, AVG, MAX, MIN 같은 집계 함수는 그룹별 결과를 계산할 때 자주 함께 쓰입니다.
- WHERE는 GROUP BY보다 먼저 실행되어, 그룹을 만들기 전 개별 행을 필터링합니다.
- HAVING은 GROUP BY 이후에 만들어진 그룹 결과를 조건으로 걸러낼 때 사용합니다.
- 그래서 “급여가 3000 이상인 직원만 모아서 부서별 평균”은 WHERE, “평균 급여가 5000 이상인 부서만 출력”은 HAVING입니다.
- SQLD에서도 SELECT 처리 순서와 WHERE/HAVING 구분은 자주 나오는 포인트입니다.
- 실무식으로 이해하면 WHERE는 재료 고르기, GROUP BY는 재료 묶기, HAVING은 완성된 묶음 중 조건에 맞는 것만 고르기입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 집계 함수 조건은 보통 HAVING에 작성합니다.
- SQLD 포인트: SELECT절에 집계 함수가 아닌 컬럼이 나오면 보통 GROUP BY에 포함되어야 합니다.
- 실행 흐름 감각: WHERE → GROUP BY → HAVING → SELECT → ORDER BY 순서로 이해하면 문제 풀이가 쉬워집니다.

3. OX 문제
WHERE절은 그룹화되기 전 개별 행을 먼저 걸러내고, HAVING절은 GROUP BY로 묶인 그룹에 대한 조건을 걸 때 사용한다.

오늘 주제: SQL/DB - OUTER JOIN의 ON 조건과 WHERE 조건 차이

1. 핵심 요약
- LEFT OUTER JOIN은 왼쪽 테이블의 행을 모두 보존하는 조인입니다.
- 오른쪽 테이블에 매칭되는 행이 없으면 오른쪽 컬럼들은 NULL로 채워집니다.
- ON절은 “어떤 행끼리 조인할지”를 정하는 조건입니다.
- WHERE절은 조인이 끝난 뒤 “최종 결과에서 어떤 행을 남길지”를 거르는 조건입니다.
- 그래서 LEFT JOIN 후 WHERE절에 오른쪽 테이블 컬럼 조건을 쓰면 NULL 확장된 행이 제거될 수 있습니다.

- 왼쪽 행은 보존하면서 오른쪽 테이블의 매칭 조건만 제한하려면 조건을 ON절에 두는 방식을 자주 고려합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: LEFT OUTER JOIN은 왼쪽 행 보존, 매칭 실패 시 오른쪽 컬럼 NULL.
- SQLD 포인트: ON절은 조인 조건, WHERE절은 조인 후 필터링이라는 처리 의미를 구분.
- 함정 포인트: 오른쪽 테이블 조건을 WHERE에 두면 OUTER JOIN 결과가 INNER JOIN처럼 줄어들 수 있음.

LEFT OUTER JOIN에서 오른쪽 테이블의 컬럼 조건을 WHERE절에 두면, 조인 후 NULL로 확장된 행이 조건에서 걸러져 결과가 INNER JOIN처럼 줄어들 수 있다.
