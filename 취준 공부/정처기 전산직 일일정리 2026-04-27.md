오늘 주제: 자료구조 - 스택과 큐

1. 핵심 요약
- 스택(Stack)은 데이터를 한쪽 끝에서만 넣고 빼는 선형 자료구조입니다.
- 가장 나중에 들어온 데이터가 가장 먼저 나가므로 후입선출, 즉 LIFO 구조라고 합니다.
- 접시를 쌓았다가 위에서부터 꺼내는 모습을 떠올리면 이해하기 쉽습니다.
- 큐(Queue)는 한쪽에서는 삽입, 반대쪽에서는 삭제가 일어나는 선형 자료구조입니다.
- 가장 먼저 들어온 데이터가 가장 먼저 나가므로 선입선출, 즉 FIFO 구조입니다.
- 줄 서기처럼 먼저 온 요청을 먼저 처리해야 하는 상황에 잘 맞습니다.
- 스택은 함수 호출, 재귀, 괄호 검사, 뒤로 가기 기능에 자주 쓰입니다.
- 큐는 작업 대기열, 프린터 출력 순서, 버퍼, BFS 탐색에서 자주 활용됩니다.

2. 자주 나오는 포인트
- 스택: LIFO, push/pop, top 개념을 묻는 문제가 자주 나옵니다.
- 큐: FIFO, enqueue/dequeue, front/rear 개념을 구분해야 합니다.
- BFS는 큐, DFS는 스택 또는 재귀와 연결해서 기억하면 좋습니다.

3. OX 문제
스택은 후입선출(LIFO) 구조라서 재귀 호출, 함수 호출 복귀 주소 관리, 괄호 검사에 활용되고, 큐는 선입선출(FIFO) 구조라서 작업 대기열이나 BFS 탐색에 활용된다.

오늘 주제: 프로그래밍 기초 - 시간복잡도와 빅오 표기법

1. 핵심 요약
- 시간복잡도는 입력 크기 n이 커질 때 알고리즘의 수행 시간이 얼마나 빠르게 증가하는지 나타내는 개념입니다.
- 시험에서는 실제 초 단위 시간보다 “연산 횟수의 증가 추세”를 보는 것이 핵심입니다.
- 빅오 표기법 O( )은 보통 최악의 경우를 기준으로 증가 차수를 표현합니다.
- 예를 들어 O(1)은 입력 크기와 거의 무관하게 일정하고, O(n)은 입력이 2배면 연산도 대체로 2배 늘어납니다.
- O(n²)은 이중 반복문처럼 입력이 커질수록 연산량이 제곱 수준으로 증가하는 경우에 자주 나옵니다.
- 빅오에서는 상수 계수와 낮은 차수 항을 생략합니다. 그래서 3n² + 10n + 5도 보통 O(n²)로 봅니다.
- 이유는 n이 매우 커지면 가장 높은 차수의 항이 전체 증가량을 지배하기 때문입니다.
- 공기업 전산직에서는 정렬 알고리즘, 탐색, 반복문 코드 분석과 함께 자주 출제됩니다.

2. 자주 나오는 포인트
- 단일 반복문은 보통 O(n), 이중 반복문은 보통 O(n²)입니다.
- 이진 탐색처럼 탐색 범위를 절반씩 줄이면 O(log n)입니다.
- 빅오는 정확한 실행 시간이 아니라 입력 증가에 따른 “증가 차수”를 나타냅니다.

3. OX 문제
빅오 표기법 O(n²)은 입력 크기 n이 커질 때 수행 시간이 대체로 n의 제곱에 비례해 증가하는 알고리즘의 증가 차수를 나타내며, 실제 실행 시간이 항상 정확히 n²초라는 뜻은 아니다.

오늘 주제: NCS 정보통신 - 다중화 방식

1. 핵심 요약
- 다중화는 하나의 전송 매체를 여러 통신 채널이 함께 사용하도록 만드는 기술입니다.
- 회선이나 주파수 자원은 한정되어 있으므로, 여러 데이터를 효율적으로 실어 보내기 위해 사용합니다.
- 대표 방식에는 주파수분할 다중화(FDM), 시분할 다중화(TDM), 파장분할 다중화(WDM), 코드분할 다중화(CDM)가 있습니다.
- FDM은 전체 주파수 대역을 여러 개로 나누어 각 신호에 서로 다른 주파수 대역을 배정합니다.
- TDM은 시간을 잘게 나누고, 각 채널이 정해진 시간 슬롯에 번갈아 데이터를 보냅니다.
- WDM은 광통신에서 서로 다른 빛의 파장을 이용해 여러 신호를 동시에 전송하는 방식입니다.
- CDM은 각 사용자에게 고유한 코드를 부여해 같은 주파수 대역을 공유하게 하는 방식입니다.
- 시험에서는 “무엇을 나누는가?”를 기준으로 구분하면 헷갈림이 줄어듭니다.

2. 자주 나오는 포인트
- FDM: 주파수 분할, 아날로그 통신·방송 계열과 자주 연결
- TDM: 시간 슬롯 분할, 디지털 전송에서 자주 출제
- WDM: 광섬유 통신, 파장 분할이 핵심 키워드

3. OX 문제
시분할 다중화(TDM)는 여러 신호가 같은 전송 매체를 사용할 때 시간 슬롯을 나누어 번갈아 전송하는 방식이다.

오늘 주제: SQL/DB - INNER JOIN과 OUTER JOIN

1. 핵심 요약
- JOIN은 두 개 이상의 테이블을 공통 컬럼이나 조건을 기준으로 연결해 하나의 결과처럼 조회하는 방법입니다.
- INNER JOIN은 조인 조건을 만족하는 행만 가져옵니다. 쉽게 말해 두 테이블의 “매칭되는 데이터”만 결과에 남습니다.
- OUTER JOIN은 조건에 맞지 않는 행도 보존할 수 있는 조인입니다.
- LEFT OUTER JOIN은 왼쪽 테이블의 행을 모두 남기고, 오른쪽 테이블에 매칭되는 값이 없으면 NULL로 채웁니다.
- RIGHT OUTER JOIN은 오른쪽 테이블을 기준으로 모두 남기고, FULL OUTER JOIN은 양쪽 모두의 미매칭 행까지 남깁니다.
- SQLD에서도 JOIN은 SELECT 문 핵심 파트로 자주 나옵니다. 특히 INNER JOIN과 OUTER JOIN의 결과 행 차이를 묻는 문제가 많습니다.
- JOIN 조건을 빠뜨리면 의도치 않은 카티션 곱이 발생해 행 수가 폭증할 수 있습니다.
- 시험에서는 “매칭되는 것만?”인지 “한쪽 기준으로 다 남기는지?”를 먼저 판단하면 헷갈림이 줄어듭니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: INNER JOIN은 조건을 만족하는 행만 출력합니다.
- SQLD 포인트: LEFT OUTER JOIN에서 오른쪽 테이블 값이 없으면 해당 컬럼은 NULL로 표시됩니다.
- 실무/시험 공통: JOIN 조건 누락 시 불필요한 조합 결과가 생길 수 있습니다.

3. OX 문제
INNER JOIN은 두 테이블에서 조인 조건을 만족하는 행만 결과에 포함하며, 조건에 맞지 않는 한쪽 테이블의 행은 결과에서 제외된다.

오늘 주제: 네트워크 - DNS와 이름 해석

1. 핵심 요약
- DNS는 Domain Name System의 약자로, 사람이 기억하기 쉬운 도메인 이름을 실제 통신에 필요한 IP 주소로 바꿔 주는 시스템입니다.
- 예를 들어 `www.example.com`에 접속하면 컴퓨터는 결국 해당 서버의 IP 주소를 알아야 패킷을 보낼 수 있습니다.
- DNS는 하나의 중앙 서버가 전부 처리하는 방식이 아니라, 루트 DNS → TLD DNS → 권한 있는 DNS 서버처럼 계층적으로 동작합니다.
- 루트 DNS는 `.com`, `.kr` 같은 최상위 도메인 서버 위치를 알려 주고, TLD DNS는 해당 도메인을 관리하는 서버를 안내합니다.
- 권한 있는 DNS 서버는 최종적으로 특정 도메인에 대응하는 IP 주소 같은 레코드를 제공합니다.
- 자주 쓰는 DNS 결과는 캐시에 저장됩니다. 그래서 같은 사이트에 다시 접속할 때 매번 전체 조회 과정을 반복하지 않아도 됩니다.
- DNS 레코드에는 IPv4 주소를 나타내는 A 레코드, IPv6 주소를 나타내는 AAAA 레코드, 별칭을 나타내는 CNAME, 메일 서버를 나타내는 MX 등이 있습니다.
- 시험에서는 “도메인 이름 ↔ IP 주소 변환”, “계층적 분산 구조”, “캐싱”, “주요 레코드 종류”를 함께 묶어 기억하면 좋습니다.

2. 자주 나오는 포인트
- DNS는 도메인 이름을 IP 주소로 변환하는 이름 해석 시스템입니다.
- 루트 DNS, TLD DNS, 권한 있는 DNS 서버로 이어지는 계층 구조를 사용합니다.
- 캐싱은 DNS 조회 속도를 높이고 네트워크 부하를 줄이지만, TTL 동안 오래된 정보가 남을 수도 있습니다.

3. OX 문제
DNS는 사람이 읽기 쉬운 도메인 이름을 IP 주소로 변환해 주는 시스템이며, 일반적으로 계층적 구조와 캐싱을 이용해 조회 효율을 높인다.

오늘 주제: SQL/DB - 트랜잭션과 원자성

1. 핵심 요약
- 트랜잭션은 데이터베이스에서 하나의 논리적 작업 단위입니다.
- 예를 들어 계좌 이체는 “A 계좌 출금”과 “B 계좌 입금”이 함께 성공해야 하나의 트랜잭션입니다.
- 트랜잭션의 대표 성질은 ACID입니다: 원자성, 일관성, 고립성, 지속성.
- 원자성은 트랜잭션 내 작업이 전부 성공하거나 전부 실패해야 한다는 성질입니다.
- 중간에 오류가 나면 일부 결과만 남기지 않고 ROLLBACK으로 이전 상태로 되돌려야 합니다.
- 반대로 모든 작업이 정상 완료되면 COMMIT을 통해 변경 내용을 확정합니다.
- SQLD에서도 트랜잭션, COMMIT, ROLLBACK, TCL 개념은 자주 나옵니다.
- 핵심은 “부분 성공을 허용하지 않는다”는 점입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: COMMIT은 변경 확정, ROLLBACK은 변경 취소입니다.
- 원자성은 “All or Nothing”으로 기억하면 좋습니다.
- 트랜잭션 제어어 TCL에는 COMMIT, ROLLBACK, SAVEPOINT가 포함됩니다.

3. OX 문제
트랜잭션의 원자성은 트랜잭션 내 연산이 모두 수행되거나 모두 수행되지 않아야 함을 의미하며, 중간에 오류가 발생하면 이미 수행된 변경도 되돌릴 수 있어야 한다.

오늘 주제: SQL/DB - 인덱스의 개념과 장단점

1. 핵심 요약
- 인덱스는 테이블에서 원하는 데이터를 빠르게 찾기 위한 자료구조입니다.
- 책 뒤의 “찾아보기”처럼, 특정 컬럼 값을 기준으로 데이터 위치를 빠르게 찾게 도와줍니다.
- 인덱스가 없으면 DBMS가 테이블 전체를 처음부터 끝까지 확인하는 Full Table Scan을 할 수 있습니다.
- 인덱스가 잘 잡혀 있으면 WHERE 조건, JOIN 조건, ORDER BY, GROUP BY 처리에서 검색 범위를 줄일 수 있습니다.
- 보통 B-Tree 인덱스가 자주 쓰이며, 범위 검색과 정렬된 탐색에 유리합니다.
- 하지만 인덱스는 별도 저장 공간을 차지하고, 데이터가 변경될 때 인덱스도 함께 갱신해야 합니다.
- 그래서 INSERT, UPDATE, DELETE가 많은 테이블에 인덱스를 너무 많이 만들면 오히려 성능이 떨어질 수 있습니다.
- SQLD에서도 자주 보는 포인트는 “인덱스는 조회 성능을 높일 수 있지만 무조건 많을수록 좋은 것은 아니다”입니다.
- 선택도가 높은 컬럼, 즉 값의 종류가 다양해서 조건으로 걸었을 때 결과 행을 많이 줄일 수 있는 컬럼에 인덱스를 두는 것이 유리합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 인덱스는 검색 속도 향상 목적이지만 저장 공간과 갱신 비용이 발생합니다.
- SQLD 포인트: WHERE, JOIN, ORDER BY에 자주 쓰이는 컬럼이 인덱스 후보가 될 수 있습니다.
- 시험 함정: 인덱스가 많으면 SELECT는 빨라질 수 있지만, DML 성능은 나빠질 수 있습니다.

3. OX 문제
인덱스는 검색 조건에 맞는 행을 더 빠르게 찾도록 도와줄 수 있지만, INSERT·UPDATE·DELETE가 자주 발생하는 테이블에서는 인덱스 유지 비용 때문에 성능 부담이 생길 수 있다.

오늘 주제: 네트워크 - IP 주소와 서브넷 마스크

1. 핵심 요약

- IPv4는 보통 `192.168.0.10`처럼 10진수 4개로 표현하지만, 내부적으로는 32비트입니다.

- 예를 들어 `/24`는 앞 24비트가 네트워크 부분이고, 나머지 8비트가 호스트 부분이라는 뜻입니다.

- 그래서 IP 주소와 서브넷 마스크를 함께 봐야 “같은 네트워크인지, 다른 네트워크인지” 판단할 수 있습니다.
- 시험에서는 `/24`, `/26` 같은 CIDR 표기와 사용 가능한 호스트 수 계산이 자주 나옵니다.

2. 자주 나오는 포인트
- 서브넷 마스크: 네트워크 부분과 호스트 부분을 구분하는 값
- 같은 서브넷 여부: IP 주소에 서브넷 마스크를 적용한 네트워크 주소가 같은지 확인
- 호스트 수 계산: 호스트 비트가 n개면 사용 가능 호스트 수는 보통 `2ⁿ - 2`

서브넷 마스크는 IP 주소에서 네트워크 부분과 호스트 부분을 구분하는 기준이며, 같은 서브넷에 속한 장비들은 일반적으로 라우터를 거치지 않고 직접 통신할 수 있다.

오늘 주제: SQL/DB - GROUP BY와 HAVING

1. 핵심 요약
- GROUP BY는 여러 행을 특정 컬럼 기준으로 묶어 “그룹 단위”로 집계할 때 사용합니다.
- 예를 들어 부서별 평균 급여, 상품별 판매 수량, 회원별 주문 횟수처럼 “~별”이라는 말이 나오면 GROUP BY를 떠올리면 됩니다.
- COUNT, SUM, AVG, MAX, MIN 같은 집계 함수는 그룹별 결과를 계산할 때 자주 함께 쓰입니다.
- WHERE는 GROUP BY보다 먼저 실행되어, 그룹을 만들기 전 개별 행을 필터링합니다.
- HAVING은 GROUP BY 이후에 만들어진 그룹 결과를 조건으로 걸러낼 때 사용합니다.
- 그래서 “급여가 3000 이상인 직원만 모아서 부서별 평균”은 WHERE, “평균 급여가 5000 이상인 부서만 출력”은 HAVING입니다.
- SQLD에서도 SELECT 처리 순서와 WHERE/HAVING 구분은 자주 나오는 포인트입니다.
- 실무식으로 이해하면 WHERE는 재료 고르기, GROUP BY는 재료 묶기, HAVING은 완성된 묶음 중 조건에 맞는 것만 고르기입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 집계 함수 조건은 보통 HAVING에 작성합니다.
- SQLD 포인트: SELECT절에 집계 함수가 아닌 컬럼이 나오면 보통 GROUP BY에 포함되어야 합니다.
- 실행 흐름 감각: WHERE → GROUP BY → HAVING → SELECT → ORDER BY 순서로 이해하면 문제 풀이가 쉬워집니다.

3. OX 문제
WHERE절은 그룹화되기 전 개별 행을 먼저 걸러내고, HAVING절은 GROUP BY로 묶인 그룹에 대한 조건을 걸 때 사용한다.
