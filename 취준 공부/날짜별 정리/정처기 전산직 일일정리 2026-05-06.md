오늘 주제: 네트워크 - TCP 신뢰성 제어와 ACK/재전송

1. 핵심 요약
- TCP는 전송 계층의 연결형 프로토콜로, 데이터를 “믿고 받을 수 있게” 보내는 데 초점이 있습니다.
- IP는 패킷을 목적지까지 보내려 노력하지만, 손실·중복·순서 뒤바뀜까지 완벽히 책임지지는 않습니다.
- TCP는 각 데이터에 순서 번호를 붙여 수신 측이 데이터의 순서를 맞출 수 있게 합니다.
- 수신 측은 데이터를 받으면 ACK, 즉 확인 응답을 보내 “여기까지 잘 받았다”는 신호를 줍니다.
- 송신 측은 일정 시간 안에 ACK를 받지 못하면 데이터가 손실됐다고 보고 재전송할 수 있습니다.
- 그래서 TCP는 순서 제어, 오류 제어, 재전송, 흐름 제어, 혼잡 제어를 함께 사용해 신뢰성을 높입니다.
- 반대로 UDP는 이런 연결 설정과 ACK 기반 재전송을 기본 제공하지 않아 빠르지만 신뢰성 보장은 약합니다.
- 시험에서는 TCP의 신뢰성 기능을 “순서 번호 + ACK + 재전송”으로 묶어 이해하면 좋습니다.

2. 자주 나오는 포인트
- TCP: 연결형, 신뢰성 제공, 순서 보장, ACK/재전송 사용
- UDP: 비연결형, 오버헤드 작음, 신뢰성·순서 보장 없음
- 흐름 제어는 수신자 처리 능력 보호, 혼잡 제어는 네트워크 혼잡 완화 목적입니다.

3. OX 문제
TCP는 순서 번호와 확인 응답(ACK), 재전송 타이머 등을 이용해 손실되거나 순서가 바뀐 데이터를 신뢰성 있게 전달하도록 돕는다.

오늘 주제: SQL/DB - CASE 표현식과 조건 분기

1. 핵심 요약
- CASE 표현식은 SQL에서 조건에 따라 다른 값을 반환하는 조건 분기 표현식입니다.
- 기본 형태는 `CASE WHEN 조건 THEN 값 ELSE 값 END`입니다.
- 조건은 위에서부터 차례로 평가되고, 처음 참이 되는 WHEN의 THEN 값이 결과가 됩니다.
- 어떤 조건도 만족하지 않으면 ELSE 값이 반환되며, ELSE를 생략하면 NULL이 될 수 있습니다.
- SELECT 절에서 코드값을 사람이 읽기 쉬운 이름으로 바꾸거나, 점수 구간을 등급으로 나눌 때 자주 씁니다.
- ORDER BY 절에서도 CASE를 써서 특정 상태를 먼저 정렬하는 식의 사용자 정의 정렬이 가능합니다.
- GROUP BY나 집계 함수와 함께 쓰면 조건별 집계도 만들 수 있습니다.
- SQLD에서도 자주 보는 포인트는 “CASE는 IF문처럼 절차를 실행하는 문장이 아니라, 조건에 따라 하나의 값을 만들어 내는 표현식”이라는 점입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: `WHEN → THEN → ELSE → END` 구조와 ELSE 생략 시 NULL 가능성을 기억하세요.
- SQLD 포인트: SELECT, ORDER BY, 집계식 안에서도 CASE를 사용할 수 있습니다.
- 주의: CASE 결과는 조건별로 “값”을 반환하는 것이지, SQL 문장을 통째로 실행 분기하는 구조는 아닙니다.

3. OX 문제
SQL에서 CASE 표현식은 조건에 따라 다른 값을 반환할 수 있으며, SELECT 절뿐 아니라 ORDER BY 절에서도 사용할 수 있다.

오늘 주제: SQL/DB - 데이터베이스 회복 REDO와 UNDO

1. 핵심 요약
- 데이터베이스 회복은 장애가 발생했을 때 DB를 일관된 상태로 되돌리는 기능입니다.
- 핵심 기준은 트랜잭션이 COMMIT 되었는지 여부입니다.
- COMMIT된 트랜잭션은 결과가 반드시 보존되어야 하므로 지속성(Durability)과 연결됩니다.
- 장애 시 일부 변경이 디스크에 완전히 반영되지 않았을 수 있어, 로그를 보고 다시 적용하는 작업이 필요합니다.
- 이를 REDO라고 하며, “완료된 작업을 다시 반영”하는 개념입니다.
- 반대로 COMMIT되지 않은 트랜잭션의 중간 변경은 남아 있으면 안 됩니다.
- 이때 로그를 이용해 변경 전 값으로 되돌리는 작업이 UNDO입니다.
- SQLD에서도 트랜잭션, COMMIT/ROLLBACK, ACID를 이해할 때 함께 연결되는 포인트입니다.
- 정처기에서는 로그, 체크포인트, REDO/UNDO 방향을 헷갈리게 자주 냅니다.

2. 자주 나오는 포인트
- 정처기: REDO는 COMMIT된 트랜잭션 재반영, UNDO는 미완료 트랜잭션 취소.
- SQLD 공통: COMMIT은 변경 확정, ROLLBACK은 미확정 변경 취소.
- ACID 연결: REDO는 지속성, UNDO는 원자성과 특히 관련이 깊습니다.

3. OX 문제
데이터베이스 회복에서 REDO는 장애 전에 COMMIT된 트랜잭션의 변경 사항을 로그를 이용해 다시 반영하고, UNDO는 COMMIT되지 않은 트랜잭션의 변경을 취소하는 작업이다.
O / X?

1. 핵심 요약
- HTTP 상태 코드는 클라이언트 요청에 대해 서버가 처리 결과를 숫자로 알려 주는 응답 코드입니다.
- 2xx는 성공, 3xx는 리다이렉션, 4xx는 클라이언트 오류, 5xx는 서버 오류 계열입니다.
- 예를 들어 200 OK는 요청 성공, 404 Not Found는 요청한 자원을 서버가 찾지 못했다는 뜻입니다.
- 4xx는 URL 오타, 권한 부족, 잘못된 요청처럼 클라이언트 요청 쪽 문제가 중심입니다.
- 5xx는 서버 내부 오류, 과부하, 게이트웨이 문제처럼 서버가 요청을 정상 처리하지 못한 경우입니다.
- HTTP는 요청과 응답 구조로 동작하며, 상태 코드는 응답 메시지의 시작 줄에 포함됩니다.
- 브라우저가 웹 페이지를 요청하면 서버는 HTML, 이미지, JSON 같은 본문과 함께 상태 코드를 돌려줍니다.
- 시험에서는 개별 코드 암기보다 “어느 계열의 문제인가”를 먼저 구분하면 헷갈림이 줄어듭니다.

2. 자주 나오는 포인트
- 200 OK: 정상 처리, 301/302: 다른 위치로 이동, 404: 자원 없음.
- 401은 인증 필요, 403은 인증 여부와 별개로 접근 권한 없음으로 구분합니다.
- 500은 서버 내부 오류, 503은 서버 과부하나 점검 등 일시적 사용 불가로 자주 나옵니다.

오늘 주제: SQL/DB - 서브쿼리 비교 연산자 ALL과 ANY

1. 핵심 요약
- 서브쿼리는 SQL문 안에 포함된 또 다른 SELECT문입니다.
- ALL과 ANY는 서브쿼리가 여러 값을 반환할 때, 바깥 쿼리의 값과 그 결과들을 비교하는 연산자입니다.
- ALL은 “서브쿼리 결과의 모든 값”과 비교 조건을 만족해야 TRUE입니다.
- 예를 들어 `급여 > ALL (10번 부서 급여들)`은 10번 부서의 모든 급여보다 커야 참입니다.
- ANY 또는 SOME은 “하나라도” 비교 조건을 만족하면 TRUE입니다.
- 예를 들어 `급여 > ANY (...)`는 결과 중 최소 하나보다만 커도 참이 될 수 있습니다.
- SQLD에서도 자주 보는 포인트는 ALL/ANY/IN/EXISTS의 의미 차이입니다.
- IN은 값 목록에 포함되는지, EXISTS는 서브쿼리 결과 행이 존재하는지를 봅니다.
- 헷갈릴 때는 ALL=전부 통과, ANY=하나만 통과로 잡으면 좋습니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: `> ALL`은 서브쿼리 결과의 최댓값보다 큰 경우와 비슷하게 해석할 수 있습니다.
- 정처기/SQLD 공통: `> ANY`는 서브쿼리 결과의 최솟값보다 큰 경우와 비슷하게 해석할 수 있습니다.
- 비교 포인트: ALL=모든 값, ANY/SOME=하나 이상, IN=포함 여부, EXISTS=행 존재 여부.

3. OX 문제
SQL에서 ALL 연산자는 서브쿼리가 반환한 모든 값과 비교 조건을 만족해야 TRUE가 된다.

오늘 주제: SQL/DB - 제2정규형과 부분 함수 종속

1. 핵심 요약
- 제2정규형(2NF)은 1NF를 만족한 상태에서 부분 함수 종속을 제거한 정규형입니다.
- 부분 함수 종속은 기본키가 복합키일 때, 일반 속성이 기본키 전체가 아니라 일부 속성에만 종속되는 경우입니다.
- 예를 들어 기본키가 (학번, 과목코드)인데 학생이름이 학번만으로 결정된다면 부분 종속입니다.
- 이 구조를 그대로 두면 같은 학생이 여러 과목을 수강할 때 학생이름이 반복 저장됩니다.
- 반복 저장은 갱신 이상, 삽입 이상, 삭제 이상으로 이어질 수 있습니다.
- 그래서 학생 정보와 수강 정보를 분리해 중복과 이상현상을 줄이는 것이 2NF의 핵심입니다.
- 단일 속성 기본키에서는 “기본키의 일부”가 없으므로 보통 부분 함수 종속 문제가 발생하지 않습니다.
- SQLD에서도 자주 보는 포인트는 1NF=원자값, 2NF=부분 함수 종속 제거, 3NF=이행 함수 종속 제거의 구분입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 2NF는 복합키에서 일부 키에만 종속되는 일반 속성을 분리한다.
- 정처기/SQLD 공통: 부분 함수 종속은 정규화 대상이며, 중복과 이상현상을 줄이기 위한 개념이다.
- 비교 포인트: 2NF는 부분 종속 제거, 3NF는 이행 종속 제거를 묻는 문제가 자주 나온다.

3. OX 문제
제2정규형(2NF)은 기본키가 복합키일 때, 기본키의 일부에만 종속되는 일반 속성을 분리해 부분 함수 종속을 제거하는 정규형이다.
