오늘 주제: SQL/DB - 외래키 참조 동작 ON DELETE CASCADE

1. 핵심 요약
- 외래키는 자식 테이블의 값이 부모 테이블의 기본키나 유일키를 참조하도록 하는 제약조건입니다.
- 목적은 존재하지 않는 부모를 참조하는 잘못된 데이터를 막아 참조 무결성을 지키는 것입니다.
- 예를 들어 주문 테이블의 고객ID는 고객 테이블에 실제 존재하는 고객ID를 참조해야 합니다.
- 부모 행을 삭제하려 할 때, 자식 행이 남아 있으면 참조 무결성 문제가 생길 수 있습니다.
- 이때 ON DELETE CASCADE는 부모 행 삭제 시 관련 자식 행도 함께 삭제되도록 하는 동작입니다.
- 반대로 SET NULL은 자식의 외래키 값을 NULL로 바꾸고, RESTRICT/NO ACTION은 부모 삭제를 막는 방식입니다.
- SQLD에서도 자주 보는 포인트는 “외래키는 관계를 구현하고 참조 무결성을 유지한다”는 점입니다.
- CASCADE는 편리하지만 주문 이력처럼 보존해야 할 데이터까지 삭제될 수 있어 업무 규칙에 맞게 선택해야 합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: FOREIGN KEY는 참조 무결성을 위한 제약조건입니다.
- ON DELETE CASCADE: 부모 행 삭제 시 참조하는 자식 행도 연쇄 삭제될 수 있습니다.
- SET NULL, RESTRICT/NO ACTION과 CASCADE의 차이를 구분해야 합니다.

오늘 주제: 네트워크 - TCP 재전송 타이머 RTO와 RTT

1. 핵심 요약
- TCP는 데이터를 신뢰성 있게 보내기 위해 순서 번호, ACK, 체크섬, 재전송을 사용합니다.
- RTT는 Round Trip Time, 즉 데이터를 보내고 ACK가 돌아오기까지 걸리는 왕복 시간입니다.
- RTO는 Retransmission Timeout으로, “ACK를 얼마 동안 기다릴지” 정하는 재전송 타이머 값입니다.
- 송신자가 세그먼트를 보낸 뒤 RTO 안에 ACK를 받지 못하면 손실 가능성이 있다고 보고 재전송할 수 있습니다.
- 네트워크 지연은 계속 변하므로 RTO를 너무 짧게 잡으면 불필요한 재전송이 늘어납니다.
- 반대로 RTO가 너무 길면 실제 손실이 생겼을 때 복구가 늦어져 전송 효율이 떨어집니다.
- 그래서 TCP는 RTT 측정값과 지연 변동을 반영해 RTO를 조정합니다.
- UDP는 TCP처럼 ACK·재전송·순서 보장을 기본 제공하지 않으므로, 필요하면 애플리케이션이 직접 처리해야 합니다.

2. 자주 나오는 포인트
- TCP 신뢰성 제어: 순서 번호, ACK, 체크섬, 재전송 타이머를 함께 사용합니다.
- RTT는 왕복 지연 시간, RTO는 ACK 대기 후 재전송 판단 기준입니다.
- TCP는 신뢰성 제공, UDP는 기본적으로 신뢰성 보장 없음으로 비교해 기억하세요.

3. OX 문제
TCP 재전송 타이머의 기준이 되는 RTO는 네트워크 지연 변화를 반영해 조정될 수 있으며, 일정 시간 안에 ACK가 오지 않으면 손실로 보고 재전송이 발생할 수 있다.

오늘 주제: SQL/DB - GROUP BY와 SELECT 컬럼 규칙

1. 핵심 요약
- GROUP BY는 같은 값을 가진 행들을 하나의 그룹으로 묶는 기능입니다.
- COUNT, SUM, AVG, MAX, MIN 같은 집계 함수는 그룹 단위로 계산됩니다.
- SELECT 절에 집계 함수와 일반 컬럼을 함께 쓰려면, 일반 컬럼은 보통 GROUP BY에 포함되어야 합니다.
- 예를 들어 부서별 평균 급여는 `SELECT 부서번호, AVG(급여) ... GROUP BY 부서번호`처럼 씁니다.
- GROUP BY 없이 `부서번호, AVG(급여)`를 함께 조회하면 “어느 부서번호를 보여줄지”가 모호해집니다.
- WHERE는 그룹을 만들기 전 개별 행을 거르는 조건입니다.
- HAVING은 GROUP BY 이후 만들어진 그룹이나 집계 결과에 거는 조건입니다.
- SQLD에서도 자주 보는 포인트는 SELECT 논리 처리 순서와 GROUP BY에 없는 일반 컬럼 사용 가능 여부입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: SELECT의 일반 컬럼은 GROUP BY 기준과 맞아야 합니다.
- WHERE는 행 조건, HAVING은 그룹 조건입니다.
- 집계 함수는 NULL을 제외하는 경우가 많지만, COUNT(*)는 행 자체를 셉니다.

3. OX 문제
SELECT 절에 집계 함수와 일반 컬럼을 함께 사용할 때, 일반 컬럼은 보통 GROUP BY에 포함되거나 집계 기준이 명확해야 한다.

오늘 주제: SQL/DB - 외래키 NULL 허용과 선택 관계

1. 핵심 요약
- 외래키는 자식 테이블의 값이 부모 테이블의 기본키나 UNIQUE 값과 연결되도록 하는 제약조건입니다.
- 핵심 목적은 “존재하지 않는 부모를 참조하지 못하게” 해서 참조 무결성을 지키는 것입니다.
- 예를 들어 주문 테이블의 고객ID가 고객 테이블에 실제 존재하는 고객ID를 참조하게 만드는 식입니다.
- 다만 외래키라고 해서 무조건 NOT NULL이어야 하는 것은 아닙니다.
- 관계가 필수라면 외래키에 NOT NULL을 함께 걸어야 하고, 관계가 선택적이면 NULL을 허용할 수 있습니다.
- 예를 들어 사원에게 부서가 반드시 있어야 하면 부서ID는 NOT NULL 외래키가 됩니다.
- 반대로 아직 부서 배정 전인 사원을 허용한다면 부서ID 외래키는 NULL일 수 있습니다.
- SQLD에서도 자주 보는 포인트는 “참조 무결성”과 “필수/선택 관계”를 구분하는 것입니다.
- FOREIGN KEY는 참조 관계를 관리하고, NOT NULL은 값의 존재 여부를 강제한다는 역할 차이를 기억하세요.

- 정처기/SQLD 공통: FOREIGN KEY는 참조 무결성, PRIMARY KEY는 개체 무결성과 관련됩니다.
- SQLD 포인트: 선택 관계라면 외래키 NULL 허용 가능, 필수 관계라면 NOT NULL을 함께 지정합니다.

오늘 주제: 네트워크 - HTTP 캐시와 Cache-Control

1. 핵심 요약
- HTTP 캐시는 서버 응답을 클라이언트나 프록시가 저장해 두었다가 같은 자원 요청에 재사용하는 기능입니다.
- 목적은 응답 속도 향상, 네트워크 트래픽 감소, 서버 부하 감소입니다.
- 예를 들어 이미지, CSS, JS 파일처럼 자주 바뀌지 않는 자원은 캐시 효과가 큽니다.
- Cache-Control은 캐시가 응답을 어떻게 저장하고 재사용할지 지시하는 HTTP 헤더입니다.
- max-age=60은 응답을 60초 동안 신선한 상태로 보고 재사용할 수 있다는 뜻입니다.
- no-cache는 이름과 달리 “저장 금지”가 아니라, 재사용 전에 서버에 재검증하라는 의미로 자주 출제됩니다.
- no-store는 응답을 캐시에 저장하지 말라는 강한 지시입니다. 로그인 정보, 금융 정보 같은 민감 데이터에 중요합니다.
- 캐시는 빠르지만 오래된 데이터를 보여줄 수 있으므로 유효 시간과 재검증 개념을 함께 이해해야 합니다.

2. 자주 나오는 포인트
- HTTP 캐시: 응답 재사용으로 속도 향상·트래픽 감소.
- Cache-Control: max-age는 유효 시간, no-cache는 재검증, no-store는 저장 금지.
- 시험 포인트: no-cache와 no-store를 헷갈리게 내는 문제가 자주 나옵니다.

3. OX 문제
