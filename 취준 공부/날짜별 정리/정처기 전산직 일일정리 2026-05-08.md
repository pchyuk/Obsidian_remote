오늘 주제: SQL/DB - CROSS JOIN과 카티션 곱

1. 핵심 요약
- JOIN은 두 개 이상의 테이블을 연결해 하나의 결과처럼 조회하는 SQL 기능입니다.
- INNER JOIN은 조인 조건을 만족하는 행만 결과에 포함합니다.
- CROSS JOIN은 두 테이블의 모든 행 조합을 만드는 조인입니다.
- 이를 카티션 곱이라고 하며, A 테이블이 3행이고 B 테이블이 4행이면 결과는 12행입니다.
- 조인 조건 없이 테이블을 나열하면 DBMS에 따라 CROSS JOIN처럼 동작해 결과가 폭증할 수 있습니다.
- 그래서 실무와 시험 모두에서 “조인 조건 누락”은 매우 중요한 실수 포인트입니다.
- SQLD에서도 JOIN 결과 행 수, INNER/OUTER/CROSS JOIN 차이, 조인 조건의 의미를 자주 확인합니다.
- CROSS JOIN은 모든 조합이 필요할 때는 유용하지만, 의도하지 않게 발생하면 성능 문제를 만들 수 있습니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: CROSS JOIN = 두 테이블 행의 모든 조합, 결과 행 수는 보통 A행 수 × B행 수.
- INNER JOIN은 조건을 만족하는 행만, OUTER JOIN은 조건 불일치 행도 한쪽 기준으로 보존할 수 있습니다.
- 조인 조건을 빠뜨리면 예상보다 많은 행이 조회될 수 있으므로 주의해야 합니다.

3. OX 문제

오늘 주제: SQL/DB - 집계 함수 COUNT와 NULL 처리

1. 핵심 요약
- 집계 함수는 여러 행을 대상으로 하나의 결과값을 계산하는 함수입니다.
- 대표적으로 COUNT, SUM, AVG, MIN, MAX가 있습니다.
- COUNT(*)는 특정 컬럼이 아니라 “행 자체”의 개수를 셉니다.
- 그래서 어떤 컬럼 값이 NULL이어도 WHERE 조건을 통과한 행이면 COUNT(*)에 포함됩니다.
- COUNT(컬럼명)은 해당 컬럼 값이 NULL인 행은 제외하고 셉니다.
- SUM, AVG, MIN, MAX도 일반적으로 NULL 값은 계산에서 제외합니다.
- 특히 AVG는 NULL을 0으로 계산하지 않습니다. NULL이 아닌 값들의 평균만 구합니다.
- SQLD에서도 COUNT(*)와 COUNT(컬럼)의 차이, NULL의 집계 제외 여부는 자주 확인하는 포인트입니다.
- GROUP BY와 함께 쓰이면 그룹별로 집계가 계산되고, HAVING은 그 집계 결과에 조건을 걸 때 사용합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: COUNT(*)는 행 수, COUNT(컬럼)은 NULL 제외한 컬럼 값 개수입니다.
- SQLD 포인트: AVG(컬럼)은 NULL을 0으로 바꾸어 평균 내는 것이 아니라 NULL을 제외하고 계산합니다.
- 시험 포인트: WHERE는 집계 전 행 필터링, HAVING은 GROUP BY 이후 집계 결과 조건입니다.

3. OX 문제
SQL 집계 함수에서 COUNT(*)는 행의 개수를 세고, COUNT(컬럼명)은 해당 컬럼 값이 NULL인 행을 제외하고 센다.

- HTTP 메서드는 클라이언트가 서버 자원에 대해 “무엇을 하고 싶은지” 나타내는 요청 방식입니다.
- GET은 자원 조회, POST는 데이터 제출·처리 요청, PUT은 자원 전체 교체, PATCH는 일부 수정, DELETE는 삭제에 주로 쓰입니다.
- “안전한 메서드”는 서버 자원 상태를 변경하려는 의도가 없는 메서드입니다. 대표적으로 GET이 있습니다.
- “멱등성”은 같은 요청을 여러 번 보내도 서버 자원 상태가 한 번 보낸 것과 같게 유지되는 성질입니다.
- GET, PUT, DELETE는 보통 멱등으로 봅니다. 예를 들어 같은 DELETE를 여러 번 보내도 “삭제된 상태”라는 결과는 같습니다.
- POST는 같은 요청을 반복하면 주문이 여러 번 생성되는 것처럼 결과가 달라질 수 있어 일반적으로 멱등이 아닙니다.
- 단, GET 요청도 서버 로그 기록이나 조회수 증가 같은 부수 효과가 있을 수 있지만, 시험에서는 “GET의 본래 목적은 조회”라고 이해하면 됩니다.
- HTTP 문제는 상태 코드, 쿠키/세션, HTTPS와 함께 메서드 의미를 묶어서 자주 출제됩니다.

2. 자주 나오는 포인트
- GET: 조회, 안전함, 멱등성 있음.
- POST: 생성·처리 요청, 안전하지 않음, 멱등성 보장 아님.
- PUT/DELETE: 안전하지는 않지만, 같은 요청 반복 시 결과 상태가 같아 멱등으로 분류.

오늘 주제: SQL/DB - BCNF와 결정자

1. 핵심 요약
- BCNF는 Boyce-Codd Normal Form, 즉 보이스-코드 정규형입니다.
- 핵심은 “모든 결정자는 후보키여야 한다”는 조건입니다.
- 결정자는 함수 종속 X → Y에서 다른 속성 Y를 결정하는 X를 말합니다.
- 예를 들어 A 값을 알면 B 값이 항상 하나로 정해질 때 A는 B의 결정자입니다.
- 3NF는 기본적으로 이행 함수 종속을 제거하지만, 후보키가 여러 개이거나 겹치는 경우 일부 이상현상이 남을 수 있습니다.
- BCNF는 3NF보다 더 엄격해서 후보키가 아닌 결정자가 존재하면 릴레이션을 분해합니다.
- 이렇게 분해하면 삽입·삭제·갱신 이상을 줄이고 데이터 중복을 더 강하게 통제할 수 있습니다.
- SQLD에서도 자주 보는 포인트는 함수 종속, 후보키, 결정자, 정규화 단계의 차이를 연결해 이해하는 것입니다.
- 단, BCNF 분해 과정에서 모든 함수 종속이 그대로 보존되지 않을 수 있다는 점도 시험 포인트입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: BCNF 조건은 “모든 결정자가 후보키”이다.
- 3NF vs BCNF: BCNF가 더 엄격하며, 후보키가 아닌 결정자를 허용하지 않는다.
- 정규화 목적: 중복과 이상현상을 줄이는 것이며, 무조건 성능 향상만을 의미하지는 않는다.

3. OX 문제
BCNF는 모든 결정자가 후보키가 되도록 분해해, 후보키가 아닌 결정자로 인한 이상현상을 줄이는 정규형이다.

오늘 주제: SQL/DB - UNIQUE 제약조건과 NULL 허용

1. 핵심 요약
- UNIQUE 제약조건은 특정 컬럼 또는 컬럼 조합의 값이 중복되지 않도록 막는 제약조건입니다.
- 예를 들어 회원 이메일에 UNIQUE를 걸면 같은 이메일을 가진 회원이 두 명 저장되지 않게 할 수 있습니다.
- PRIMARY KEY도 중복을 허용하지 않지만, PRIMARY KEY는 테이블의 대표 식별자이며 반드시 NULL이 될 수 없습니다.
- UNIQUE는 “중복 방지”가 핵심이고, 대표 식별자인지는 별개의 문제입니다.
- NULL은 “값이 없음/알 수 없음”을 뜻하므로, UNIQUE 컬럼에서 NULL 허용 방식은 DBMS마다 세부 동작이 다를 수 있습니다.
- 그래서 시험에서는 UNIQUE와 NOT NULL을 분리해서 보는 것이 중요합니다.
- SQLD에서도 제약조건 비교 문제가 자주 나오며, PRIMARY KEY, UNIQUE, NOT NULL, FOREIGN KEY의 역할을 구분해야 합니다.
- 정리하면 PRIMARY KEY는 유일성+NOT NULL+대표 식별자, UNIQUE는 유일성 보장, NOT NULL은 NULL 금지입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: PRIMARY KEY는 중복 불가 + NULL 불가입니다.
- 정처기/SQLD 공통: UNIQUE는 중복 방지가 목적이며, NULL 허용 여부는 NOT NULL과 구분해야 합니다.
- 비교 포인트: FOREIGN KEY는 참조 무결성, CHECK는 값의 조건 제한, UNIQUE는 값의 중복 제한입니다.

3. OX 문제
UNIQUE 제약조건은 컬럼 값의 중복을 막는 제약조건이지만, PRIMARY KEY와 달리 일반적으로 NULL 허용 여부는 별도로 판단해야 한다.
