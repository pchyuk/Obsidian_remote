오늘 주제: SQL/DB - NULL의 의미와 3값 논리

1. 핵심 요약
- NULL은 0, 공백, 빈 문자열이 아니라 “값이 없음” 또는 “알 수 없음”을 뜻합니다.
- SQL에서는 TRUE/FALSE만 있는 것이 아니라 UNKNOWN까지 포함한 3값 논리를 사용합니다.
- 예를 들어 `NULL = NULL`은 참처럼 보이지만, SQL에서는 “알 수 없음”으로 처리될 수 있습니다.
- 그래서 NULL을 비교할 때는 `= NULL`이 아니라 `IS NULL`, `IS NOT NULL`을 써야 합니다.
- `WHERE`절은 조건 결과가 TRUE인 행만 통과시키므로, FALSE뿐 아니라 UNKNOWN도 제외됩니다.
- 집계 함수에서도 주의가 필요합니다. `COUNT(*)`는 전체 행 수를 세지만, `COUNT(컬럼명)`은 NULL을 제외하고 셉니다.
- SQLD에서도 자주 보는 포인트는 NULL 비교, 집계 함수와 NULL, 조건식에서 UNKNOWN 처리입니다.
- 정처기 DB 문제에서도 “NULL은 0과 같다” 같은 함정이 자주 나오니 확실히 구분해야 합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: NULL 여부는 `IS NULL`, `IS NOT NULL`로 판단한다.
- SQLD 포인트: `WHERE 컬럼 = NULL`은 올바른 NULL 검색 조건이 아니다.
- 집계 함수 포인트: `COUNT(*)`는 NULL 포함 행 수, `COUNT(컬럼)`은 해당 컬럼의 NULL 제외 개수.

3. OX 문제
SQL에서 NULL은 0이나 빈 문자열을 의미하지 않고, 비교 연산에서 TRUE/FALSE가 아닌 UNKNOWN 결과가 나올 수 있다.

1. 핵심 요약

- 후보키는 유일성과 최소성을 모두 만족하는 키입니다. 즉 행을 구분할 수 있고, 불필요한 속성을 더 포함하지 않습니다.
- 기본키는 후보키 중 대표로 선택한 키입니다. 테이블의 각 행을 식별하는 기준이 됩니다.
- 기본키는 중복될 수 없고, NULL 값을 가질 수 없습니다. 이것이 개체 무결성과 연결됩니다.
- 대체키는 후보키 중 기본키로 선택되지 않은 나머지 키입니다.
- 외래키는 다른 테이블의 기본키를 참조하는 키로, 테이블 간 관계를 표현합니다.
- SQLD에서도 엔터티 식별자, 주식별자/보조식별자, 기본키/외래키 개념이 자주 연결되어 나옵니다.
- 정처기에서는 키의 종류와 무결성 제약을 함께 묻는 문제가 많습니다.

2. 자주 나오는 포인트
- 후보키 = 유일성 + 최소성
- 기본키 = NULL 불가 + 중복 불가 + 개체 무결성
- 외래키 = 다른 테이블 참조 + 참조 무결성과 관련

3. OX 문제
기본키는 후보키 중에서 대표로 선택한 키이며, 기본키 속성은 NULL 값을 가질 수 없고 한 릴레이션 안에서 각 튜플을 유일하게 식별해야 한다.

오늘 주제: 네트워크 - HTTP와 무상태성

1. 핵심 요약
- HTTP는 웹 브라우저 같은 클라이언트와 웹 서버가 데이터를 주고받기 위한 응용 계층 프로토콜입니다.
- 기본 구조는 클라이언트가 요청(Request)을 보내고, 서버가 응답(Response)을 돌려주는 방식입니다.
- 예를 들어 브라우저가 “이 페이지 보여줘”라고 요청하면 서버는 HTML, 이미지, JSON 같은 데이터를 응답합니다.
- HTTP의 중요한 특징은 무상태성(stateless)입니다. 즉, 서버는 기본적으로 이전 요청을 자동으로 기억하지 않습니다.
- 그래서 로그인 유지, 장바구니, 사용자 설정처럼 “이전 상태”가 필요한 기능은 쿠키, 세션, 토큰 등을 함께 사용합니다.
- HTTP는 평문 통신이고, HTTPS는 HTTP에 SSL/TLS 암호화를 적용해 기밀성과 무결성을 강화한 방식입니다.
- 동작 원리는 보통 TCP 연결 위에서 요청 메시지와 응답 메시지를 주고받는 형태로 이해하면 됩니다.
- 시험에서는 HTTP가 어느 계층인지, 요청/응답 구조, 무상태성, HTTPS와의 차이가 자주 나옵니다.

2. 자주 나오는 포인트
- HTTP: 응용 계층 프로토콜, 요청/응답 방식
- 무상태성: 이전 요청 정보를 기본적으로 기억하지 않음
- HTTPS: HTTP + SSL/TLS, 암호화로 보안 강화

3. OX 문제
HTTP는 웹 클라이언트와 서버가 요청과 응답을 주고받기 위한 응용 계층 프로토콜이며, 기본적으로 각 요청은 독립적으로 처리되므로 상태 유지를 위해 쿠키나 세션 같은 보조 수단을 사용할 수 있다.

- 공개키 암호화는 공개키와 개인키라는 서로 다른 키 쌍을 사용합니다.

- 공개키 방식은 키 분배 문제를 줄일 수 있지만, 대칭키보다 연산 속도가 느립니다.

1. 핵심 요약

- 사용자는 뷰를 통해 복잡한 JOIN이나 조건식을 단순한 이름으로 재사용할 수 있습니다.
- 특정 컬럼이나 행만 보여주도록 만들 수 있어 보안 측면에서도 자주 활용됩니다.
- 예를 들어 직원 테이블에서 급여 컬럼은 숨기고 이름·부서만 보여주는 뷰를 만들 수 있습니다.
- SQLD에서도 뷰는 “독립적인 실제 테이블이 아니라 기본 테이블에 의존하는 객체”라는 점이 중요합니다.
- 기본 테이블이 변경되면 뷰 조회 결과도 영향을 받을 수 있습니다.
- 단, 집계 함수, GROUP BY, DISTINCT, 조인 등이 포함된 뷰는 INSERT/UPDATE/DELETE가 제한될 수 있습니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 뷰는 논리적 테이블이며, 보안성·편의성·논리적 독립성에 도움을 줍니다.
- 뷰는 기본 테이블에 의존하므로 기본 테이블이 삭제되거나 구조가 바뀌면 영향을 받을 수 있습니다.
- 모든 뷰가 갱신 가능한 것은 아니며, 복잡한 SELECT 기반 뷰는 갱신이 제한될 수 있습니다.

오늘 주제: 네트워크 - TCP 흐름 제어와 혼잡 제어

1. 핵심 요약

2. 자주 나오는 포인트
- 흐름 제어: 수신 버퍼 초과 방지, 수신 윈도우(rwnd), 송신자-수신자 간 속도 조절
- 혼잡 제어: 네트워크 과부하 방지, 혼잡 윈도우(cwnd), 슬로 스타트/혼잡 회피
- 시험에서는 “흐름 제어 = 수신자 기준”, “혼잡 제어 = 네트워크 상태 기준” 구분이 핵심입니다.

오늘 주제: SQL/DB - DML·DDL·DCL·TCL 명령어 분류

1. 핵심 요약
- SQL 명령어는 역할에 따라 DML, DDL, DCL, TCL로 나눕니다.
- DML은 데이터를 조회·변경하는 명령입니다. SELECT, INSERT, UPDATE, DELETE가 대표적입니다.
- DDL은 데이터베이스 구조를 정의하거나 변경합니다. CREATE, ALTER, DROP, TRUNCATE가 자주 나옵니다.
- DCL은 권한을 제어합니다. GRANT는 권한 부여, REVOKE는 권한 회수입니다.
- TCL은 트랜잭션을 제어합니다. COMMIT, ROLLBACK, SAVEPOINT가 핵심입니다.
- 동작 원리로 보면 DML은 “데이터 내용”을 다루고, DDL은 “테이블 구조”를 다룹니다.
- COMMIT 전의 DML 변경은 확정 전 상태일 수 있고, ROLLBACK으로 취소할 수 있습니다.
- 반면 DDL은 DBMS에 따라 실행 즉시 자동 COMMIT되는 경우가 많아, DML처럼 쉽게 되돌린다고 생각하면 위험합니다.
- SQLD에서도 명령어 분류, 트랜잭션 제어, COMMIT/ROLLBACK의 의미는 기초 빈출입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: SELECT는 DML, CREATE/ALTER/DROP은 DDL, GRANT/REVOKE는 DCL, COMMIT/ROLLBACK은 TCL.
- DML은 데이터 조작, DDL은 구조 정의, DCL은 권한 제어, TCL은 트랜잭션 확정·취소.
- DELETE는 DML이라 조건 삭제와 ROLLBACK 가능성을 함께 묻고, TRUNCATE는 DDL 성격으로 자주 비교됩니다.

3. OX 문제
SQL에서 COMMIT과 ROLLBACK은 트랜잭션의 변경 내용을 확정하거나 취소하는 TCL에 해당하며, INSERT·UPDATE·DELETE 같은 DML 작업 결과를 제어할 때 사용된다.
