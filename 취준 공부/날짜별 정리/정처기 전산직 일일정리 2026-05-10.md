1. 핵심 요약
- DNS는 사람이 읽기 쉬운 도메인 이름을 IP 주소나 관련 서비스 정보로 바꿔 주는 이름 해석 시스템입니다.
- DNS 레코드는 도메인에 연결된 정보를 저장하는 항목입니다.

- AAAA 레코드는 도메인 이름을 IPv6 주소에 매핑합니다.

- NS 레코드는 해당 도메인의 권한 있는 네임서버를 지정합니다.
- 동작 원리는 클라이언트가 DNS 서버에 질의하면, DNS 서버가 캐시나 계층적 조회를 통해 해당 레코드를 찾아 응답하는 방식입니다.

- A=IPv4, AAAA=IPv6, MX=메일 서버, CNAME=별칭, NS=권한 네임서버.

오늘 주제: SQL/DB - 집합 연산 INTERSECT와 EXCEPT/MINUS

1. 핵심 요약
- 집합 연산은 여러 SELECT 결과를 하나의 결과처럼 결합할 때 사용합니다.
- UNION은 두 결과를 합치고, INTERSECT는 두 결과의 공통 행만 남깁니다.
- EXCEPT 또는 Oracle 계열의 MINUS는 첫 번째 SELECT 결과에서 두 번째 SELECT 결과에 있는 행을 제외합니다.
- 집합 연산을 쓰려면 각 SELECT의 컬럼 개수가 같고, 대응되는 컬럼의 데이터 타입이 호환되어야 합니다.
- 기본적으로 UNION, INTERSECT, EXCEPT/MINUS는 중복 제거 성격을 가집니다.
- 반면 UNION ALL은 중복을 제거하지 않고 그대로 합칩니다.
- SQLD에서도 자주 보는 포인트는 “합집합·교집합·차집합” 의미를 SQL 결과 행 기준으로 해석하는 것입니다.
- ORDER BY는 보통 개별 SELECT가 아니라 전체 집합 연산 결과의 마지막에 적용합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: UNION=합집합, INTERSECT=교집합, EXCEPT/MINUS=차집합.
- SQLD 포인트: UNION과 UNION ALL의 중복 제거 여부를 구분하세요.
- 실무형 함정: 집합 연산의 양쪽 SELECT는 컬럼 수와 타입 호환성이 맞아야 합니다.

3. OX 문제
INTERSECT는 두 SELECT 결과에 공통으로 존재하는 행만 반환하고, EXCEPT 또는 MINUS는 첫 번째 SELECT 결과에서 두 번째 SELECT 결과에 존재하는 행을 제외한다.

1. 핵심 요약
- 라운드 로빈(Round Robin)은 준비 큐에 있는 프로세스들에게 CPU를 일정 시간씩 번갈아 배정하는 선점형 스케줄링입니다.
- 이때 각 프로세스가 한 번에 사용할 수 있는 시간을 시간 할당량 또는 타임 퀀텀이라고 합니다.
- 시간이 끝났는데도 프로세스가 완료되지 않으면 CPU를 빼앗기고 준비 큐의 뒤로 이동합니다.
- 그래서 긴 작업 하나가 CPU를 오래 독점하는 것을 막고, 여러 작업이 비교적 공평하게 실행될 수 있습니다.
- 시분할 시스템이나 대화형 시스템처럼 응답 시간이 중요한 환경에 잘 어울립니다.
- 다만 타임 퀀텀이 너무 작으면 프로세스 전환이 너무 자주 일어나 문맥 교환 오버헤드가 커집니다.
- 반대로 타임 퀀텀이 너무 크면 FCFS처럼 동작해 응답성이 떨어질 수 있습니다.
- 핵심은 “공평한 순환 배정”과 “타임 퀀텀 크기의 적절한 조절”입니다.

2. 자주 나오는 포인트
- 라운드 로빈은 선점형 스케줄링이며, 시간 할당량이 끝나면 CPU를 회수할 수 있습니다.
- 타임 퀀텀이 너무 작으면 문맥 교환 오버헤드가 증가합니다.
- 타임 퀀텀이 너무 크면 FCFS와 비슷해져 대화형 시스템의 장점이 약해집니다.

3. OX 문제
라운드 로빈 스케줄링은 각 프로세스에 시간 할당량을 주고 순환하며 CPU를 배정하므로, 시간 할당량이 너무 작으면 문맥 교환 오버헤드가 커질 수 있다.

오늘 주제: SQL/DB - DELETE·TRUNCATE·DROP 차이

1. 핵심 요약
- DELETE, TRUNCATE, DROP은 모두 “삭제”와 관련 있지만 삭제 대상이 다릅니다.
- DELETE는 테이블 안의 행 데이터를 삭제하는 DML입니다.
- DELETE는 WHERE 조건을 사용해 일부 행만 삭제할 수 있고, WHERE를 생략하면 모든 행이 삭제될 수 있습니다.
- TRUNCATE는 테이블 구조는 남기고 모든 행을 빠르게 비우는 명령입니다.
- TRUNCATE는 보통 WHERE 조건으로 일부 행만 지우는 용도가 아닙니다.
- DROP은 테이블이라는 객체 자체를 삭제하므로 구조와 데이터가 함께 사라집니다.
- SQLD에서도 DELETE/TRUNCATE/DROP 구분은 자주 나옵니다.
- 핵심은 DELETE=행 삭제, TRUNCATE=전체 데이터 비우기, DROP=테이블 제거로 잡으면 됩니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: DELETE는 DML, TRUNCATE·DROP은 DDL 성격으로 분류하는 문제가 자주 나옵니다.
- DELETE는 WHERE 조건 가능, TRUNCATE는 일반적으로 조건 삭제가 아니라 전체 비우기입니다.
- DROP은 테이블 구조까지 제거하므로 TRUNCATE보다 삭제 범위가 큽니다.

3. OX 문제
TRUNCATE는 테이블의 모든 행을 빠르게 제거하는 명령으로, 일반적으로 WHERE 조건으로 일부 행만 선택해 삭제하는 용도로 사용한다.

1. 핵심 요약

- 한 번 전체를 훑으면 가장 큰 값이 맨 뒤로 이동하는 식으로 정렬 범위가 줄어듭니다.
- 동작 원리가 단순해서 구현은 쉽지만, 비교와 교환을 반복하므로 효율은 낮은 편입니다.
- 평균 시간복잡도와 최악 시간복잡도는 O(n²)입니다.
- 이미 정렬된 배열에서 교환이 한 번도 없으면 종료하는 최적화를 넣으면 최선은 O(n)이 될 수 있습니다.
- 선택 정렬은 “최솟값을 찾아 교환”, 삽입 정렬은 “앞쪽 정렬 구간에 끼워 넣기”, 버블 정렬은 “인접 원소 비교·교환”으로 구분하면 좋습니다.

2. 자주 나오는 포인트
- 버블 정렬: 인접 원소 비교 후 교환, 평균/최악 O(n²)

- “항상 O(n log n)” 같은 표현은 버블 정렬에는 맞지 않습니다.

3. OX 문제
버블 정렬은 인접한 두 원소를 비교해 필요하면 교환하는 과정을 반복하는 정렬이며, 평균 시간복잡도는 O(n²)이다.

오늘 주제: SQL/DB - 윈도우 함수 PARTITION BY와 ORDER BY

1. 핵심 요약
- 윈도우 함수는 행을 유지한 상태에서 순위, 누적합, 평균 같은 분석 값을 계산하는 SQL 기능입니다.
- 기본 형태는 함수() OVER (...)이고, OVER 안에 PARTITION BY와 ORDER BY를 자주 씁니다.
- PARTITION BY는 계산 범위를 나누는 기준입니다. 예를 들어 부서별 순위를 구하면 부서가 파티션 기준입니다.
- ORDER BY는 각 파티션 안에서 어떤 순서로 계산할지 정합니다. 순위 함수나 누적합에서 특히 중요합니다.
- GROUP BY는 여러 행을 그룹별 한 행으로 줄일 수 있지만, 윈도우 함수는 원래 행 수를 유지합니다.
- 그래서 “각 사원의 정보 + 부서 내 급여 순위”처럼 상세 행과 분석 결과를 함께 볼 때 유용합니다.
- SQLD에서도 자주 보는 포인트는 ROW_NUMBER, RANK, DENSE_RANK, SUM() OVER 같은 기본 패턴입니다.
- 정처기에서는 GROUP BY와 윈도우 함수의 차이, OVER 절의 역할을 구분하는 문제가 나올 수 있습니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: GROUP BY는 행을 집계해 줄일 수 있고, 윈도우 함수는 행을 유지한 채 계산값을 붙입니다.
- PARTITION BY는 “계산 그룹 나누기”, ORDER BY는 “그 안에서 계산 순서 정하기”입니다.
- ROW_NUMBER는 동점도 고유 번호, RANK는 동점 후 순위 건너뜀, DENSE_RANK는 동점 후 순위 건너뛰지 않음입니다.

2. 자주 나오는 포인트

- TTL이 0이면 패킷은 폐기되고, ICMP 오류 메시지가 사용될 수 있다.

오늘 주제: SQL/DB - 제1정규형과 원자값

1. 핵심 요약
- 제1정규형(1NF)은 릴레이션의 각 속성 값이 원자값을 가지도록 하는 정규형입니다.
- 원자값이란 한 칸에 더 이상 의미 있게 나누기 어려운 하나의 값만 들어가는 상태입니다.
- 예를 들어 전화번호 컬럼에 `010-1111, 010-2222`처럼 여러 값을 넣으면 1NF 관점에서 좋지 않습니다.
- 이런 다중값 속성은 검색, 수정, 삭제 때 불편하고 데이터 불일치가 생기기 쉽습니다.
- 해결 방법은 전화번호를 별도 행으로 나누거나, 학생-전화번호 같은 별도 테이블로 분리하는 것입니다.
- 정규화는 단순히 테이블을 쪼개는 암기가 아니라 중복과 이상현상을 줄이기 위한 설계 과정입니다.
- SQLD에서도 데이터 모델링과 정규화 문제에서 1NF=원자값, 반복 속성 제거를 자주 확인합니다.
- 이후 2NF는 부분 함수 종속 제거, 3NF는 이행 함수 종속 제거로 이어집니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 1NF는 반복 그룹 제거와 원자값 보장이 핵심입니다.
- 한 컬럼에 여러 값을 콤마로 저장하는 방식은 조회·수정·무결성 관리에 불리합니다.
- 1NF → 2NF → 3NF 흐름을 “원자값 → 부분 종속 제거 → 이행 종속 제거”로 정리하세요.

오늘 주제: 네트워크 - 라우팅 테이블과 최장 접두사 일치

1. 핵심 요약
- 라우팅 테이블은 라우터가 목적지 IP 패킷을 어디로 보낼지 결정할 때 참고하는 경로 목록입니다.
- 보통 목적지 네트워크, 서브넷 마스크/프리픽스, 다음 홉, 인터페이스, 비용 정보 등을 포함합니다.
- 라우터는 패킷의 목적지 IP를 보고 라우팅 테이블에서 일치하는 경로를 찾습니다.
- 이때 여러 경로가 동시에 맞을 수 있는데, 더 구체적인 경로를 우선 선택합니다.
- 이를 최장 접두사 일치(Longest Prefix Match)라고 합니다.
- 예를 들어 10.1.2.3은 10.0.0.0/8에도 속하고 10.1.2.0/24에도 속하지만, /24가 더 구체적이므로 우선됩니다.
- 기본 경로 0.0.0.0/0은 모든 목적지와 일치하지만 가장 덜 구체적인 경로입니다.
- 그래서 다른 경로가 없을 때 “일단 여기로 보내라”는 기본 게이트웨이 역할로 자주 사용됩니다.
- 시험에서는 “라우터는 목적지 IP와 라우팅 테이블을 보고 다음 홉을 결정한다”는 흐름을 먼저 잡으면 좋습니다.

2. 자주 나오는 포인트
- 라우팅 테이블: 목적지 네트워크별 다음 홉/출구 인터페이스를 저장합니다.
- 최장 접두사 일치: 여러 경로가 매칭되면 프리픽스 길이가 더 긴, 더 구체적인 경로를 우선합니다.
- 기본 경로: 0.0.0.0/0은 구체 경로가 없을 때 사용되는 마지막 후보입니다.
