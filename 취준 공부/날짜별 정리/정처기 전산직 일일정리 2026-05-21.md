오늘 주제: 네트워크 - IPv4 헤더 TTL과 Protocol 필드

1. 핵심 요약
- IPv4 헤더는 IP 패킷이 목적지까지 전달되도록 돕는 제어 정보 묶음입니다.
- TTL(Time To Live)은 패킷의 수명을 제한하는 필드입니다.
- 실제 동작에서는 라우터를 한 홉 지날 때마다 TTL이 보통 1씩 감소합니다.
- TTL이 0이 되면 라우터는 패킷을 폐기해 라우팅 루프가 무한히 지속되는 것을 막습니다.
- Protocol 필드는 IP 패킷 안에 실린 상위 계층 프로토콜을 구분합니다.
- 예를 들어 TCP, UDP, ICMP 중 무엇으로 넘겨 처리해야 하는지 알려 줍니다.
- 즉 IP는 목적지까지 패킷을 보내고, Protocol 값은 받은 데이터를 어느 프로토콜 처리부로 넘길지 알려 주는 역할입니다.
- Header Checksum은 IPv4 헤더 오류 검출에 쓰이고, Identification·Flags·Fragment Offset은 단편화와 재조립에 관련됩니다.
- 시험에서는 TTL을 “시간”보다 “홉 수 제한”처럼 이해하는 것이 안전합니다.

2. 자주 나오는 포인트
- TTL: 라우터 통과 시 감소, 0이면 폐기, 라우팅 루프 방지.
- Protocol 필드: TCP/UDP/ICMP 등 상위 프로토콜 식별.
- IPv4 단편화 관련 필드: Identification, Flags, Fragment Offset.

오늘 주제: SQL/DB - WHERE와 HAVING의 처리 시점

1. 핵심 요약
- WHERE는 그룹화되기 전, 개별 행을 먼저 걸러내는 조건절입니다.
- HAVING은 GROUP BY로 그룹을 만든 뒤, 그룹이나 집계 결과에 조건을 거는 절입니다.
- 예를 들어 “급여가 300만 원 이상인 사원만 집계”는 WHERE 급여 >= 3000000에 가깝습니다.
- 반대로 “부서별 평균 급여가 300만 원 이상인 부서만 조회”는 HAVING AVG(급여) >= 3000000입니다.
- SQL의 논리적 처리 순서는 보통 FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY로 이해하면 좋습니다.
- 그래서 WHERE 단계에서는 아직 그룹이나 집계 결과가 만들어지지 않았다고 보면 됩니다.
- SQLD에서도 WHERE와 HAVING의 차이, GROUP BY에 없는 일반 컬럼을 SELECT에 함부로 쓰면 안 된다는 포인트가 자주 나옵니다.
- 핵심은 “행 조건은 WHERE, 그룹/집계 조건은 HAVING”입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: WHERE는 행 필터, HAVING은 그룹 필터입니다.
- GROUP BY 이후 COUNT, SUM, AVG 같은 집계 함수 조건은 주로 HAVING에서 판단합니다.
- WHERE로 미리 행을 줄이면 집계 대상 자체가 줄어들고, HAVING은 집계 후 결과 그룹을 줄입니다.

3. OX 문제
SQL에서 HAVING절은 GROUP BY로 묶인 그룹이나 집계 결과에 조건을 적용할 때 사용하며, WHERE절은 그룹화되기 전 개별 행을 먼저 필터링한다.

1. 핵심 요약

- 프로그램 전체를 처음부터 모두 적재하지 않아도 되므로 초기 실행 부담과 메모리 낭비를 줄일 수 있습니다.
- CPU가 어떤 페이지를 참조했는데 메모리에 없으면 페이지 부재(Page Fault)가 발생합니다.
- 이때 운영체제는 보조기억장치에서 해당 페이지를 찾아 메모리 프레임에 적재합니다.
- 빈 프레임이 없으면 FIFO, LRU, Clock 같은 페이지 교체 알고리즘으로 내보낼 페이지를 고릅니다.
- 요구 페이징은 지역성 원리, 즉 최근 사용한 주변 데이터가 다시 사용될 가능성이 높다는 특성을 활용합니다.
- 단, 페이지 부재가 너무 자주 발생하면 디스크 I/O가 늘어나 CPU가 실제 작업보다 페이지 교체에 시간을 쓰게 됩니다.
- 이런 상태가 심해지면 스래싱으로 이어져 전체 성능이 급격히 떨어질 수 있습니다.

2. 자주 나오는 포인트
- 요구 페이징 = “처음부터 전부 적재”가 아니라 “필요할 때 적재”
- 페이지 부재 = 참조한 페이지가 메모리에 없을 때 발생

오늘 주제: SQL/DB - ORDER BY 다중 컬럼 정렬

1. 핵심 요약
- ORDER BY는 SELECT 결과의 행 순서를 정렬하는 절입니다.
- 기본 정렬 방향은 ASC, 즉 오름차순입니다.
- DESC를 붙이면 내림차순으로 정렬합니다.
- 여러 컬럼을 지정하면 앞 컬럼이 1차 정렬 기준이 됩니다.
- 앞 컬럼 값이 같은 행들끼리는 다음 컬럼 기준으로 다시 정렬됩니다.
- 예를 들어 ORDER BY 부서번호 ASC, 급여 DESC는 부서번호는 오름차순, 같은 부서 안에서는 급여 높은 순입니다.
- SQLD에서도 ORDER BY는 SELECT 기본 문법에서 자주 확인하는 포인트입니다.
- ORDER BY를 지정하지 않으면 조회 결과 순서는 보장된다고 보면 안 됩니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: ASC는 기본값, DESC는 내림차순입니다.
- 다중 정렬: 앞 컬럼 기준이 같을 때 다음 컬럼 기준이 적용됩니다.
- ORDER BY는 결과 정렬용이지, 행을 필터링하는 WHERE나 그룹 조건 HAVING과 역할이 다릅니다.

3. OX 문제
SQL에서 ORDER BY에 여러 컬럼을 지정하면 앞 컬럼 값이 같은 행들 안에서 다음 컬럼 기준 정렬이 적용될 수 있다.

오늘 주제: 운영체제 - 프로세스 생성 fork와 부모·자식 프로세스

1. 핵심 요약
- 프로세스 생성은 운영체제가 새로운 실행 단위를 만드는 과정입니다.
- 대표적으로 Unix/Linux 계열에서는 fork 방식으로 부모 프로세스를 바탕으로 자식 프로세스를 만들 수 있습니다.

- 부모와 자식은 서로 다른 PID를 가지며, 스케줄러에 의해 독립적으로 실행될 수 있습니다.

- 프로세스는 서로 독립된 주소 공간을 갖는 것이 기본이라, 같은 프로세스 안 자원을 공유하는 스레드와 구분해야 합니다.
- 시험에서는 “부모/자식 관계가 있다”와 “같은 프로세스는 아니다”를 함께 잡는 것이 중요합니다.

2. 자주 나오는 포인트
- fork: 부모 프로세스를 바탕으로 자식 프로세스 생성
- 부모/자식 프로세스: 서로 다른 PID를 가진 별도 실행 단위
- 프로세스 vs 스레드: 프로세스는 주소 공간 독립, 스레드는 같은 프로세스 자원 공유

오늘 주제: 네트워크 - TCP와 UDP 신뢰성 비교

1. 핵심 요약
- TCP와 UDP는 모두 전송 계층 프로토콜이며, IP 위에서 애플리케이션 간 통신을 돕습니다.
- TCP는 연결형 프로토콜로, 데이터를 보내기 전에 3-way handshake로 연결을 설정합니다.
- TCP는 순서 번호, ACK, 재전송, 흐름 제어, 혼잡 제어를 통해 신뢰성 있는 전송을 제공합니다.
- UDP는 비연결형 프로토콜로, 연결 설정 없이 데이터그램 단위로 바로 전송합니다.
- UDP는 헤더가 단순하고 처리 부담이 작아 지연이 적을 수 있습니다.
- 대신 UDP 자체는 데이터 도착 보장, 순서 보장, 손실 재전송을 기본 제공하지 않습니다.
- 그래서 영상 스트리밍, 음성 통화, DNS처럼 빠른 응답이 중요한 곳에서 UDP가 자주 쓰입니다.
- 단, UDP에서도 신뢰성이 필요하면 애플리케이션 계층에서 별도 로직을 구현해야 합니다.

2. 자주 나오는 포인트
- TCP: 연결형, 신뢰성, 순서 보장, ACK/재전송, 흐름·혼잡 제어
- UDP: 비연결형, 낮은 오버헤드, 빠른 전송, 신뢰성 직접 보장 없음
- 시험에서는 “UDP가 TCP보다 항상 신뢰성이 높다” 같은 문장은 틀린 표현입니다.

3. OX 문제
UDP는 연결 설정 과정이 없고 오버헤드가 작을 수 있지만, TCP처럼 순서 보장·손실 재전송·흐름 제어를 기본 기능으로 제공하지는 않는다.
