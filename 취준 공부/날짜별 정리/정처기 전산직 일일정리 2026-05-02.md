오늘 주제: SQL/DB - CHECK와 NOT NULL 제약조건

1. 핵심 요약
- 제약조건은 테이블에 잘못된 데이터가 들어가지 않도록 DBMS가 강제로 지켜 주는 규칙입니다.
- CHECK는 컬럼 값이 특정 조건을 만족하는지 검사하는 제약조건입니다.
- 예를 들어 `나이 >= 0`, `점수 BETWEEN 0 AND 100`, `상태 IN ('정상', '중지')` 같은 업무 규칙을 DB에 걸 수 있습니다.
- NOT NULL은 해당 컬럼에 NULL 값이 들어가지 못하게 합니다.
- NULL은 0이나 빈 문자열이 아니라 “값이 없음/알 수 없음”을 뜻하므로, 필수 입력 항목에는 NOT NULL을 붙입니다.
- 동작 원리는 INSERT나 UPDATE가 실행될 때 DBMS가 제약조건을 검사하고, 조건을 위반하면 저장을 거부하는 방식입니다.
- SQLD에서도 제약조건은 자주 나옵니다. 특히 PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL, CHECK의 역할 구분이 중요합니다.
- 정처기에서는 무결성 관점에서 자주 연결됩니다. CHECK는 도메인 무결성, FOREIGN KEY는 참조 무결성과 관련이 깊습니다.

2. 자주 나오는 포인트
- CHECK: 값의 범위나 조건을 제한한다. 예: 점수는 0~100 사이.
- NOT NULL: 필수 입력 컬럼에 NULL 저장을 막는다.
- 정처기/SQLD 공통: PRIMARY KEY는 기본적으로 UNIQUE + NOT NULL 성격을 가진다.

3. OX 문제
CHECK 제약조건은 컬럼 값이 지정한 조건을 만족하도록 제한할 수 있으며, NOT NULL은 해당 컬럼에 NULL 값이 저장되지 않도록 막는 제약조건이다.

오늘 주제: 소프트웨어공학 - 자료 흐름도 DFD

1. 핵심 요약
- 자료 흐름도(DFD, Data Flow Diagram)는 시스템 안에서 데이터가 어떻게 이동하고 처리되는지 표현하는 분석 도구입니다.
- “누가 데이터를 입력하는가 → 어떤 처리를 거치는가 → 어디에 저장되는가 → 어디로 출력되는가”를 그림으로 정리합니다.
- 주요 구성 요소는 프로세스, 데이터 흐름, 데이터 저장소, 단말/외부 엔터티입니다.
- 프로세스는 입력 데이터를 출력 데이터로 변환하는 처리 기능을 의미합니다.
- 데이터 저장소는 파일, DB 테이블처럼 데이터가 보관되는 장소입니다.
- 외부 엔터티는 시스템 밖에서 데이터를 주고받는 사람, 조직, 다른 시스템입니다.
- DFD는 프로그램 실행 순서나 조건 분기보다 데이터의 이동과 변환에 초점을 둡니다.
- 보통 상위 수준의 문맥도에서 시작해 하위 수준으로 점진적으로 상세화합니다.

2. 자주 나오는 포인트
- DFD 구성 요소: 프로세스, 데이터 흐름, 데이터 저장소, 외부 엔터티.
- DFD는 제어 흐름도가 아니라 자료 흐름 중심의 모델링 도구입니다.
- 자료 사전(DD)은 DFD에 나온 데이터 흐름과 저장소의 세부 구조를 정의합니다.

오늘 주제: 네트워크 - TCP/IP 캡슐화와 역캡슐화

1. 핵심 요약
- 캡슐화는 송신 측에서 상위 계층의 데이터에 각 계층의 제어 정보를 붙여 하위 계층으로 내려보내는 과정입니다.
- 응용 계층의 데이터가 전송 계층으로 내려가면 TCP/UDP 헤더가 붙고, 인터넷 계층에서는 IP 헤더가 붙습니다.
- 네트워크 접근 계층에서는 MAC 주소 등 링크 계층 정보가 붙어 프레임 단위로 전송됩니다.
- 즉 데이터는 계층을 내려가면서 “데이터 → 세그먼트/데이터그램 → 패킷 → 프레임”처럼 형태가 바뀝니다.
- 수신 측에서는 반대로 하위 계층부터 헤더를 확인하고 제거하면서 상위 계층으로 올려보내는데, 이를 역캡슐화라고 합니다.
- 각 헤더에는 목적지 주소, 출발지 주소, 포트 번호, 오류 확인 등 해당 계층이 맡은 기능을 수행하기 위한 정보가 들어갑니다.
- 계층 구조를 쓰면 각 계층이 자기 역할에 집중할 수 있어 통신 과정을 표준화하고 문제 원인을 나눠서 파악하기 쉽습니다.
- 시험에서는 OSI/TCP-IP 계층, PDU 명칭, TCP/UDP 헤더와 IP 헤더의 역할 구분이 자주 연결됩니다.

2. 자주 나오는 포인트
- TCP/UDP 포트 번호는 전송 계층, IP 주소는 인터넷/네트워크 계층, MAC 주소는 데이터링크 계층 쪽 개념입니다.
- 송신은 캡슐화, 수신은 역캡슐화 순서로 이해하면 됩니다.
- PDU 명칭: 전송 계층은 세그먼트(TCP) 또는 데이터그램(UDP), 인터넷 계층은 패킷, 데이터링크 계층은 프레임으로 자주 출제됩니다.

3. OX 문제
TCP/IP 통신에서 캡슐화는 상위 계층의 데이터에 각 계층의 헤더를 붙여 하위 계층으로 내려보내는 과정이고, 수신 측에서는 이를 역순으로 제거하며 해석한다.

오늘 주제: SQL/DB - 트랜잭션 ACID와 지속성

1. 핵심 요약
- 트랜잭션은 데이터베이스에서 하나의 논리적 작업 단위입니다.
- 예를 들어 계좌 이체는 “A 계좌 출금”과 “B 계좌 입금”이 함께 성공해야 하나의 트랜잭션으로 볼 수 있습니다.
- ACID는 트랜잭션이 안전하게 처리되기 위한 네 가지 성질입니다.
- 원자성은 모두 수행되거나 모두 취소되어야 한다는 성질입니다.
- 일관성은 트랜잭션 전후에도 데이터베이스의 제약조건과 규칙이 깨지지 않아야 한다는 뜻입니다.
- 격리성은 동시에 여러 트랜잭션이 실행되어도 서로 간섭하지 않는 것처럼 처리되어야 한다는 성질입니다.
- 지속성은 COMMIT된 결과가 장애가 발생하더라도 보존되어야 한다는 뜻입니다.
- SQLD에서도 트랜잭션, COMMIT, ROLLBACK, ACID의 의미는 자주 확인하는 기본 포인트입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: ACID 네 글자와 의미를 정확히 연결해야 합니다.
- COMMIT은 변경 확정, ROLLBACK은 변경 취소입니다.
- 지속성은 “확정된 결과의 보존”, 원자성은 “전부 성공 또는 전부 실패”로 구분하세요.

3. OX 문제
트랜잭션의 ACID 성질 중 지속성은 COMMIT된 변경 내용이 장애가 발생하더라도 보존되어야 한다는 의미이다.

2. 자주 나오는 포인트
- 해시 테이블의 평균 검색 시간은 O(1), 충돌이 많으면 성능이 떨어질 수 있음
- 충돌 해결: 체이닝, 선형 조사 등 개방 주소법
- 해시 함수는 키를 저장 위치로 변환하는 함수

오늘 주제: 네트워크 - HTTPS와 TLS의 역할

1. 핵심 요약
- HTTPS는 HTTP에 TLS를 적용해 안전하게 통신하는 방식입니다.
- HTTP는 웹 요청·응답을 주고받는 프로토콜이고, TLS는 그 데이터를 암호화해 보호합니다.
- 즉 HTTPS는 “HTTP + TLS”로 이해하면 됩니다.
- TLS는 전송 중인 데이터가 도청되지 않도록 기밀성을 제공합니다.
- 또한 중간에서 데이터가 바뀌었는지 확인할 수 있게 무결성도 제공합니다.
- 서버 인증서를 통해 사용자가 접속한 서버가 진짜 서버인지 확인하는 인증 기능도 합니다.
- 실제 데이터 전송은 빠른 대칭키 암호화를 사용하고, 그 대칭키를 안전하게 정하는 과정에 공개키 기반 기술이 활용됩니다.
- 시험에서는 HTTPS가 HTTP보다 안전한 이유를 “암호화, 무결성, 인증” 세 단어로 정리하면 좋습니다.
- 기본 포트는 HTTP 80번, HTTPS 443번입니다.

2. 자주 나오는 포인트
- HTTPS = HTTP 자체가 사라지는 것이 아니라, HTTP 통신을 TLS로 보호하는 방식
- TLS의 핵심 기능: 기밀성, 무결성, 서버 인증
- 포트 번호 구분: HTTP 80 / HTTPS 443

3. OX 문제
HTTPS는 HTTP 메시지를 TLS로 보호해 전송 구간의 도청과 변조 위험을 줄이며, 일반적으로 서버 인증서를 이용해 서버 신원을 확인한다.  
O / X?

오늘 주제: 프로그래밍 기초 - 포인터와 메모리 주소

1. 핵심 요약
- 포인터는 “값”이 아니라 “메모리 주소”를 저장하는 변수입니다.
- 예를 들어 일반 변수 a가 실제 데이터 10을 담는다면, 포인터 p는 a가 저장된 위치의 주소를 담을 수 있습니다.
- C 언어 기준으로 &a는 a의 주소를 구하는 연산이고, *p는 p가 가리키는 주소의 값을 접근하는 역참조입니다.
- 포인터를 쓰면 배열, 문자열, 동적 메모리, 함수 인자 전달을 효율적으로 다룰 수 있습니다.
- 하지만 초기화하지 않은 포인터, NULL 포인터 역참조, 이미 해제한 메모리를 다시 참조하는 dangling pointer는 오류를 만들 수 있습니다.
- 특히 포인터는 메모리에 직접 접근하므로 잘못 쓰면 프로그램 비정상 종료나 보안 취약점으로 이어질 수 있습니다.
- 시험에서는 포인터의 개념 자체보다 주소 연산자, 역참조, NULL, 배열과 포인터 관계를 자주 묻습니다.

2. 자주 나오는 포인트
- 포인터 변수는 데이터 값 자체가 아니라 메모리 주소를 저장한다.
- &는 주소를 구할 때, *는 선언 또는 역참조 문맥에서 쓰인다.
- 잘못된 포인터 접근은 런타임 오류, 메모리 손상, 보안 문제로 연결될 수 있다.

3. OX 문제
포인터 변수는 값 자체를 저장하는 일반 변수와 달리 다른 변수나 메모리 공간의 주소를 저장하며, 잘못 사용하면 잘못된 메모리 접근 문제가 발생할 수 있다.

오늘 주제: SQL/DB - DML UPDATE와 WHERE 조건

1. 핵심 요약
- DML은 테이블의 데이터를 조회·삽입·수정·삭제하는 명령어입니다.
- 대표적으로 SELECT, INSERT, UPDATE, DELETE가 DML에 해당합니다.
- UPDATE는 이미 존재하는 행의 컬럼 값을 수정할 때 사용합니다.
- 기본 형태는 `UPDATE 테이블명 SET 컬럼명 = 값 WHERE 조건;`입니다.
- WHERE절은 “어떤 행을 수정할지”를 제한하는 조건입니다.
- WHERE절을 생략하면 조건 필터가 없으므로 테이블의 모든 행이 수정될 수 있습니다.
- 그래서 UPDATE와 DELETE는 시험에서도 WHERE절 유무를 자주 묻습니다.
- SQLD에서도 DML 분류, 트랜잭션 제어 가능 여부, UPDATE 문법은 기본 빈출 포인트입니다.
- UPDATE 같은 DML 작업은 보통 COMMIT 전까지 ROLLBACK으로 취소할 수 있는 트랜잭션 대상입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: INSERT·UPDATE·DELETE는 DML, CREATE·ALTER·DROP은 DDL입니다.
- UPDATE에서 WHERE절을 생략하면 모든 행이 수정될 수 있습니다.
- DML 결과는 트랜잭션 제어, 즉 COMMIT·ROLLBACK과 연결해서 자주 출제됩니다.

3. OX 문제
SQL에서 UPDATE 문은 기존 행의 값을 수정하는 DML이며, WHERE 조건을 생략하면 테이블의 모든 행이 수정될 수 있다.
