오늘 주제: 네트워크 - TCP 연결 종료 4-way handshake

1. 핵심 요약
- TCP는 연결형 프로토콜이라 데이터를 주고받기 전 연결을 설정하고, 통신이 끝나면 연결을 종료합니다.
- 연결 종료는 보통 4-way handshake라고 부르며 FIN과 ACK를 주고받는 절차입니다.
- FIN은 “나는 더 이상 보낼 데이터가 없다”는 종료 요청입니다.
- ACK는 “네 종료 요청을 확인했다”는 응답입니다.
- TCP는 양방향 통신이므로 한쪽 송신만 닫혔다고 전체 연결이 바로 끝나는 것은 아닙니다.
- 예를 들어 A가 FIN을 보내면 B는 ACK로 확인하고, B도 보낼 데이터를 마치면 FIN을 보냅니다.
- 마지막으로 A가 ACK를 보내면 종료 절차가 마무리됩니다.
- 시험에서는 연결 설정은 3-way handshake, 연결 종료는 4-way handshake로 구분하는 것이 중요합니다.
- UDP는 연결 설정·종료 절차 없이 데이터그램을 보내므로 TCP와 비교해서 자주 출제됩니다.

2. 자주 나오는 포인트
- TCP 설정: SYN → SYN+ACK → ACK, 즉 3-way handshake
- TCP 종료: FIN → ACK → FIN → ACK, 즉 4-way handshake
- TCP는 연결형·신뢰성·순서 보장, UDP는 비연결형·낮은 오버헤드 중심

3. OX 문제
TCP 연결 종료 과정은 보통 FIN과 ACK를 주고받는 4-way handshake로 이루어지며, 연결을 끊는 쪽과 상대방이 각각 자신의 송신 종료를 확인한다.

오늘 주제: SQL/DB - GROUP BY와 HAVING 집계 조건

1. 핵심 요약
- GROUP BY는 같은 값을 가진 행들을 하나의 그룹으로 묶는 SQL 절입니다.
- 보통 COUNT, SUM, AVG, MAX, MIN 같은 집계 함수와 함께 사용합니다.
- 예를 들어 “부서별 평균 급여”는 부서번호로 GROUP BY 한 뒤 AVG(급여)를 계산하는 구조입니다.
- WHERE는 그룹을 만들기 전, 개별 행을 먼저 걸러냅니다.
- HAVING은 GROUP BY 이후 만들어진 그룹이나 집계 결과에 조건을 겁니다.
- 그래서 “급여가 300만 원 이상인 사원만”은 WHERE, “평균 급여가 300만 원 이상인 부서만”은 HAVING에 가깝습니다.
- SQLD에서도 SELECT 처리 순서와 WHERE/HAVING 차이는 자주 보는 포인트입니다.
- GROUP BY를 사용할 때 SELECT에는 보통 그룹 기준 컬럼이나 집계 함수가 와야 한다는 점도 중요합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: WHERE는 행 조건, HAVING은 그룹 조건입니다.
- GROUP BY 없이도 전체 테이블에 대한 COUNT, AVG 같은 집계는 가능할 수 있습니다.
- GROUP BY 컬럼이 아닌 일반 컬럼을 SELECT에 무작정 쓰면 SQL 오류 또는 부정확한 의미가 될 수 있습니다.

3. OX 문제
SQL의 GROUP BY는 지정한 컬럼 값이 같은 행들을 그룹으로 묶고, HAVING은 그룹화 이후 집계 결과에 조건을 적용할 때 사용한다.

오늘 주제: 네트워크 - IP 단편화와 MTU

1. 핵심 요약
- MTU는 Maximum Transmission Unit의 약자로, 한 링크에서 한 번에 전송할 수 있는 최대 데이터 크기입니다.
- IP 패킷이 지나가는 경로의 MTU보다 크면 그대로 통과하지 못할 수 있습니다.
- IPv4에서는 조건에 따라 큰 패킷을 여러 조각으로 나누는 단편화가 발생할 수 있습니다.
- 각 단편에는 원래 패킷을 식별하기 위한 ID, 위치를 나타내는 Fragment Offset, 뒤에 단편이 더 있는지 나타내는 MF 비트가 사용됩니다.
- 조각난 단편들은 중간 라우터가 아니라 최종 수신 호스트에서 다시 조립됩니다.
- 단편 중 하나라도 손실되면 원래 패킷 전체를 복원하기 어려워 재전송 부담이 커질 수 있습니다.
- 그래서 단편화는 가능하면 피하는 것이 성능과 안정성 측면에서 좋습니다.
- IPv6에서는 라우터가 단편화를 하지 않고, 송신 측이 Path MTU Discovery로 적절한 크기를 찾는 방식이 중요합니다.

2. 자주 나오는 포인트
- MTU: 링크에서 한 번에 보낼 수 있는 최대 전송 단위
- IPv4 단편화: 큰 패킷을 여러 조각으로 나누고, 최종 수신 측에서 재조립
- IPv6 비교: 라우터 단편화 없음, 송신 측이 경로 MTU를 고려

3. OX 문제
IPv4에서 전송하려는 패킷 크기가 경로의 MTU보다 크면 단편화가 발생할 수 있으며, 수신 측은 조각난 단편들을 다시 조립한다.

- SJF는 Shortest Job First의 약자로, 실행 시간이 가장 짧을 것으로 예상되는 프로세스를 먼저 실행합니다.
- 짧은 작업을 먼저 처리하면 전체 평균 대기 시간을 줄이는 데 유리합니다.
- 다만 실제 실행 시간을 미리 정확히 알기 어렵다는 한계가 있습니다.
- 실행 시간이 긴 프로세스는 짧은 작업에 계속 밀려 기아 현상이 생길 수 있습니다.
- 기본 SJF는 보통 비선점형으로 설명됩니다.
- SRT는 Shortest Remaining Time으로, 남은 실행 시간이 더 짧은 프로세스가 오면 CPU를 빼앗을 수 있는 선점형 방식입니다.
- 시험에서는 SJF와 SRT를 “비선점/선점” 관점으로 구분하는 문제가 자주 나옵니다.

2. 자주 나오는 포인트
- SJF: 평균 대기 시간 감소에 유리하지만 실행 시간 예측이 어렵다.
- SJF의 단점: 긴 작업이 계속 밀리는 기아 현상 가능.
- SRT: SJF의 선점형 변형으로, “남은 실행 시간”이 기준이다.

3. OX 문제
SJF 스케줄링은 실행 시간이 짧을 것으로 예상되는 프로세스를 먼저 선택하는 방식이며, 선점형으로 구현한 변형은 SRT 방식이라고 볼 수 있다.

2. 자주 나오는 포인트
- 패리티·체크섬·CRC = 오류 검출 기법으로 분류합니다.

오늘 주제: 자료구조 - 이진 탐색 트리 BST

1. 핵심 요약
- 이진 탐색 트리(BST)는 각 노드가 최대 두 개의 자식을 가지는 이진 트리의 한 종류입니다.
- 핵심 규칙은 “왼쪽 서브트리 값 < 현재 노드 값 < 오른쪽 서브트리 값”입니다.
- 이 규칙 덕분에 값을 찾을 때 전체를 다 보지 않고, 현재 노드와 비교해 왼쪽 또는 오른쪽 한쪽만 탐색할 수 있습니다.
- 예를 들어 찾는 값이 현재 노드보다 작으면 오른쪽은 볼 필요 없이 왼쪽으로 내려갑니다.
- 트리의 균형이 잘 잡혀 있으면 탐색, 삽입, 삭제가 평균적으로 O(log n)에 가깝습니다.
- 하지만 값이 정렬된 순서로 계속 삽입되면 한쪽으로 치우쳐 연결 리스트처럼 변할 수 있고, 이때는 O(n)까지 느려질 수 있습니다.
- BST를 중위 순회하면 값이 오름차순으로 출력된다는 점도 자주 나옵니다.

2. 자주 나오는 포인트
- BST 규칙: 왼쪽은 작고, 오른쪽은 크다.
- 중위 순회 결과는 오름차순 정렬 형태가 된다.
- 균형 트리는 O(log n), 편향 트리는 O(n)까지 성능 저하 가능.

3. OX 문제
이진 탐색 트리에서는 어떤 노드의 왼쪽 서브트리에 있는 모든 값이 그 노드보다 작고, 오른쪽 서브트리에 있는 모든 값이 그 노드보다 크다는 규칙을 이용해 탐색 범위를 줄일 수 있다.

오늘 주제: SQL/DB - DDL CREATE·ALTER와 스키마 변경

1. 핵심 요약
- DDL은 Data Definition Language, 즉 데이터베이스 객체의 구조를 정의하거나 변경하는 명령어입니다.
- 대표 명령어는 CREATE, ALTER, DROP, TRUNCATE 등이 있습니다.
- CREATE는 테이블·뷰·인덱스 같은 객체를 새로 만드는 명령입니다.
- ALTER는 이미 존재하는 객체의 구조를 바꾸는 명령입니다.
- 예를 들어 ALTER TABLE로 컬럼 추가, 컬럼 타입 변경, 컬럼 삭제, 제약조건 추가 등을 할 수 있습니다.
- DROP은 객체 자체를 제거하므로 테이블을 DROP하면 구조와 데이터가 함께 사라집니다.
- SQLD에서도 DDL/DML/DCL/TCL 분류는 자주 나옵니다.
- DML은 INSERT·UPDATE·DELETE처럼 데이터를 조작하고, DCL은 GRANT·REVOKE처럼 권한을 제어합니다.
- TCL은 COMMIT·ROLLBACK처럼 트랜잭션 결과를 확정하거나 취소합니다.
- 시험에서는 “데이터를 바꾸는가, 구조를 바꾸는가, 권한을 바꾸는가, 트랜잭션을 제어하는가”로 구분하면 실수가 줄어듭니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: CREATE=생성, ALTER=구조 변경, DROP=객체 삭제.
- DROP은 테이블 구조까지 제거하고, DELETE는 테이블 구조는 유지한 채 행 데이터만 삭제합니다.
- ALTER TABLE은 컬럼 추가/수정/삭제, 제약조건 추가 등에 사용될 수 있습니다.

3. OX 문제
ALTER TABLE은 기존 테이블의 구조를 변경하는 DDL이며, 컬럼 추가·수정·삭제나 제약조건 추가 등에 사용할 수 있다.

오늘 주제: SQL/DB - 윈도우 함수 순위 ROW_NUMBER·RANK·DENSE_RANK

1. 핵심 요약
- 윈도우 함수는 행을 그룹으로 합쳐 줄이지 않고, 각 행을 유지한 채 분석 값을 계산하는 SQL 기능입니다.
- 기본 형태는 함수명() OVER (PARTITION BY ... ORDER BY ...)처럼 씁니다.
- PARTITION BY는 계산 범위를 나누는 기준이고, ORDER BY는 그 범위 안에서 순서를 정하는 기준입니다.
- ROW_NUMBER는 정렬된 행마다 1, 2, 3처럼 고유한 번호를 붙입니다.
- RANK는 동점이면 같은 순위를 주고, 다음 순위는 동점자 수만큼 건너뜁니다.
- DENSE_RANK는 동점이면 같은 순위를 주지만, 다음 순위를 건너뛰지 않습니다.
- 예를 들어 점수가 100, 90, 90, 80이면 RANK는 1, 2, 2, 4이고 DENSE_RANK는 1, 2, 2, 3입니다.
- SQLD에서도 자주 보는 포인트는 “GROUP BY는 행 수가 줄 수 있지만, 윈도우 함수는 원래 행을 유지한다”는 차이입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: ROW_NUMBER, RANK, DENSE_RANK의 동점 처리 차이를 구분하세요.
- SQLD 포인트: OVER 절의 ORDER BY가 순위 계산 기준이 됩니다.
- 함정 포인트: 윈도우 함수는 집계 함수처럼 보이지만 결과 행을 반드시 하나로 합치는 것은 아닙니다.

3. OX 문제
ROW_NUMBER, RANK, DENSE_RANK는 모두 윈도우 함수로 순위를 계산할 수 있지만, 동점이 있을 때 ROW_NUMBER는 행마다 고유 번호를 부여하고 RANK는 다음 순위를 건너뛰며 DENSE_RANK는 순위를 건너뛰지 않는다.
