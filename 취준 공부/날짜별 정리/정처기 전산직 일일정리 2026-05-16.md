오늘 주제: 네트워크 - ARP 캐시와 ARP 스푸핑

1. 핵심 요약
- ARP는 같은 LAN에서 IP 주소에 대응하는 MAC 주소를 알아내는 프로토콜입니다.
- 예를 들어 192.168.0.5에게 보내야 하는데 MAC 주소를 모르면 ARP Request를 브로드캐스트합니다.
- 해당 IP를 가진 장비가 ARP Reply로 자신의 MAC 주소를 알려줍니다.
- ARP 캐시는 이렇게 알아낸 IP-MAC 매핑을 일정 시간 저장하는 표입니다.
- 매번 브로드캐스트를 보내지 않아도 되므로 네트워크 부담을 줄일 수 있습니다.
- 다만 장비 교체, IP 변경, 가상 IP 이동 등으로 MAC이 바뀌면 오래된 캐시 때문에 통신이 안 될 수 있습니다.
- ARP는 기본적으로 응답을 강하게 검증하지 않아, 공격자가 거짓 매핑을 주입할 수 있습니다.
- 이를 ARP 스푸핑 또는 ARP 포이즈닝이라고 하며, 중간자 공격이나 트래픽 가로채기에 악용될 수 있습니다.
- 핵심 흐름은 “IP로 최종 목적지를 찾고, MAC으로 같은 구간의 다음 전달 대상을 찾는다”입니다.

2. 자주 나오는 포인트
- ARP: IP 주소 → MAC 주소 해석, 같은 네트워크 구간에서 주로 동작
- ARP 캐시: 반복 조회와 브로드캐스트 감소, 단 잘못된 캐시는 통신 장애 가능
- ARP 스푸핑: 거짓 IP-MAC 매핑으로 트래픽을 가로채거나 우회시키는 공격

3. OX 문제
ARP 캐시는 한 번 학습한 IP-MAC 매핑을 일정 시간 저장해 반복 조회를 줄이지만, 오래되거나 잘못된 매핑이 남으면 통신 장애나 보안 문제가 생길 수 있다.

오늘 주제: SQL/DB - EXISTS 서브쿼리와 존재 여부 판단

1. 핵심 요약
- EXISTS는 서브쿼리의 “결과 행이 존재하는가?”를 판단하는 조건입니다.
- 서브쿼리가 한 행이라도 반환하면 TRUE, 한 행도 반환하지 않으면 FALSE가 됩니다.
- SELECT 1, SELECT *, SELECT 컬럼명처럼 SELECT 목록에 무엇을 쓰는지는 핵심이 아닙니다.
- 중요한 것은 서브쿼리의 WHERE 조건을 만족하는 행이 실제로 존재하느냐입니다.
- 상관 서브쿼리와 함께 자주 쓰이며, 바깥 쿼리의 각 행마다 관련 데이터가 있는지 확인할 수 있습니다.
- 예를 들어 “주문이 있는 고객만 조회”할 때 고객별로 주문 테이블에 관련 행이 존재하는지 검사할 수 있습니다.
- SQLD에서도 IN, EXISTS, NOT EXISTS 차이를 자주 묻습니다.
- IN은 값 목록과 비교하는 느낌이 강하고, EXISTS는 조건을 만족하는 행의 존재 여부를 봅니다.
- 특히 NULL이 섞일 수 있는 NOT IN보다 NOT EXISTS가 더 안전하게 해석되는 경우가 있습니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: EXISTS는 결과값 자체보다 “행 존재 여부”가 핵심입니다.
- SQLD 포인트: SELECT 목록보다 서브쿼리의 FROM/WHERE 조건이 참·거짓 판단에 중요합니다.
- 비교 포인트: IN=값 비교, EXISTS=존재 여부, 상관 서브쿼리=바깥 행 값을 참조 가능.

3. OX 문제
SQL에서 EXISTS 서브쿼리는 서브쿼리 결과 행의 존재 여부를 판단하므로, 서브쿼리의 SELECT 목록에 어떤 컬럼을 적었는지가 조건의 참/거짓을 결정하는 핵심이다.

오늘 주제: SQL/DB - 관계형 모델 선택과 투영

1. 핵심 요약
- 관계형 모델에서 릴레이션은 테이블, 튜플은 행, 속성은 컬럼에 대응해서 이해하면 좋습니다.
- 선택(selection)은 조건을 만족하는 튜플, 즉 “행”을 고르는 연산입니다.
- SQL에서는 WHERE 절이 선택 연산에 가깝습니다. 예: WHERE deptno = 10
- 투영(projection)은 필요한 속성, 즉 “컬럼”을 고르는 연산입니다.
- SQL에서는 SELECT 절의 컬럼 목록이 투영 연산에 가깝습니다. 예: SELECT name, salary
- 헷갈리는 포인트는 SQL의 SELECT라는 단어가 관계대수의 selection과 이름이 비슷하다는 점입니다.
- SQLD에서도 자주 보는 포인트는 “SELECT 절은 컬럼 결정, WHERE 절은 행 필터링”이라는 구분입니다.
- 중복 제거가 필요하면 SELECT DISTINCT를 사용하며, 이는 투영 결과에서 중복 튜플을 줄이는 개념과 연결됩니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: WHERE = 조건에 맞는 행 필터링, SELECT 컬럼 목록 = 필요한 컬럼 선택.
- 관계형 모델 용어: 릴레이션=테이블, 튜플=행, 속성=컬럼, 도메인=속성이 가질 수 있는 값의 범위.
- SQLD 포인트: SELECT DISTINCT는 조회 결과의 중복을 제거하지만, 원본 테이블 데이터를 삭제하는 것은 아닙니다.

3. OX 문제
SQL에서 WHERE는 조건에 맞는 행을 고르는 선택 연산에 가깝고, SELECT 절의 컬럼 목록은 필요한 속성을 고르는 투영 연산에 가깝다.

오늘 주제: SQL/DB - EXISTS 서브쿼리와 행 존재 여부

- NOT EXISTS는 반대로 조건을 만족하는 행이 없을 때 TRUE가 됩니다.
- SQLD에서도 EXISTS/IN/NOT IN/NOT EXISTS 차이는 자주 나옵니다.
- 특히 NOT IN은 서브쿼리 결과에 NULL이 섞이면 예상과 다르게 결과가 안 나올 수 있어 주의해야 합니다.

- 정처기/SQLD 공통: EXISTS는 값 비교보다 “행 존재 여부”가 핵심입니다.
- SQLD 포인트: IN은 목록 값 비교, EXISTS는 관련 행 존재 확인으로 구분하세요.

오늘 주제: 네트워크 - UDP와 비연결형 데이터그램 전송

1. 핵심 요약
- UDP는 User Datagram Protocol의 약자로, 전송 계층에서 동작하는 비연결형 프로토콜입니다.
- 비연결형이란 TCP처럼 3-way handshake로 연결을 먼저 맺지 않고 바로 데이터를 보낸다는 뜻입니다.
- UDP의 전송 단위는 데이터그램이며, 각 데이터그램은 비교적 독립적으로 전달됩니다.
- 헤더가 단순해 오버헤드가 작고, 연결 설정 시간이 없어 지연이 적은 편입니다.
- 대신 UDP 자체는 순서 보장, 손실 재전송, 흐름 제어, 혼잡 제어를 TCP처럼 기본 제공하지 않습니다.
- 그래서 일부 데이터가 손실되더라도 빠른 전달이 중요한 DNS, VoIP, 스트리밍, 온라인 게임 등에 활용됩니다.
- 신뢰성이 필요하면 애플리케이션 계층에서 ACK, 재전송, 순서 번호 등을 별도로 설계해야 합니다.
- 시험에서는 TCP와 UDP를 “신뢰성 vs 속도/단순성” 관점으로 비교하면 이해가 쉽습니다.

2. 자주 나오는 포인트
- TCP: 연결형, 신뢰성, 순서 보장, ACK/재전송, 흐름·혼잡 제어 제공.
- UDP: 비연결형, 데이터그램, 낮은 오버헤드, 빠른 전송, 신뢰성 보장은 기본 제공 아님.
- DNS는 주로 UDP 53번을 사용하지만, 큰 응답이나 영역 전송 등에서는 TCP도 사용될 수 있습니다.

3. OX 문제
