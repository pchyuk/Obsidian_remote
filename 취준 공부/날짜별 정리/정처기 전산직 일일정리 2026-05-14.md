오늘 주제: SQL/DB - 트랜잭션 격리 수준과 동시성 이상 현상

1. 핵심 요약
- 트랜잭션 격리 수준은 여러 트랜잭션이 동시에 실행될 때 서로의 작업 결과를 얼마나 볼 수 있는지 정하는 기준입니다.
- 격리성이 높을수록 데이터 일관성은 좋아지지만, 잠금과 대기 시간이 늘어 동시성은 낮아질 수 있습니다.
- 격리성이 낮으면 동시성은 좋아질 수 있지만 Dirty Read, Non-repeatable Read, Phantom Read 같은 이상 현상이 생길 수 있습니다.
- Dirty Read는 다른 트랜잭션이 아직 COMMIT하지 않은 값을 읽는 현상입니다.
- Non-repeatable Read는 같은 행을 다시 읽었는데, 중간에 다른 트랜잭션이 수정·커밋해서 값이 달라지는 현상입니다.
- Phantom Read는 같은 조건으로 다시 조회했는데, 중간에 다른 트랜잭션이 행을 삽입·삭제해 결과 행 집합이 달라지는 현상입니다.
- SQLD에서도 트랜잭션, COMMIT/ROLLBACK, 격리 수준, 동시성 이상 현상은 자주 연결되어 나옵니다.
- 핵심은 “격리 수준 ↑ = 일관성 ↑, 동시성 ↓ / 격리 수준 ↓ = 동시성 ↑, 이상 현상 가능성 ↑”로 이해하는 것입니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: Dirty Read는 미확정 데이터 읽기, Non-repeatable Read는 같은 행 값 변경, Phantom Read는 조건 조회 결과 행 변화입니다.
- 격리성은 ACID 중 I에 해당하며, 트랜잭션 간 간섭을 줄이는 성질입니다.
- 격리 수준을 무조건 높이는 것이 항상 좋은 것은 아니며, 성능과 일관성의 균형이 중요합니다.

1. 핵심 요약
- IPv4 단편화는 패킷 크기가 경로의 MTU보다 클 때 패킷을 여러 조각으로 나누는 과정입니다.
- MTU는 한 번에 전송 가능한 최대 프레임/패킷 크기라고 이해하면 됩니다.
- 단편화된 조각들은 같은 원래 패킷에서 왔다는 것을 식별자 필드로 구분합니다.
- MF 플래그는 More Fragments의 약자로, 뒤에 단편이 더 남아 있는지를 표시합니다.
- 보통 마지막 단편은 더 이상 뒤 조각이 없으므로 MF가 0입니다.
- 단편 오프셋은 해당 조각이 원래 데이터에서 어느 위치에 있었는지를 나타냅니다.

- 단편이 하나라도 손실되면 원래 패킷 전체 재조립이 어려워질 수 있어 성능 부담이 생깁니다.
- 그래서 실제 네트워크에서는 단편화를 줄이기 위해 적절한 패킷 크기 조절이 중요합니다.

2. 자주 나오는 포인트
- MTU보다 큰 IPv4 패킷은 단편화될 수 있으며, 단편화는 오버헤드와 손실 위험을 증가시킬 수 있습니다.
- MF 플래그는 “뒤에 조각이 더 있는가?”, 단편 오프셋은 “원래 데이터의 어느 위치인가?”를 나타냅니다.

오늘 주제: SQL/DB - DEFAULT 제약조건과 NULL 처리

1. 핵심 요약
- DEFAULT는 컬럼 값을 생략했을 때 자동으로 들어갈 기본값을 정하는 제약조건입니다.
- 예를 들어 가입일 컬럼에 DEFAULT CURRENT_DATE를 두면, INSERT 시 가입일을 생략했을 때 현재 날짜가 들어갈 수 있습니다.
- 중요한 점은 “값을 생략한 경우”와 “명시적으로 NULL을 넣은 경우”를 구분해야 한다는 것입니다.
- 일반적으로 컬럼에 NULL을 직접 넣으면 DEFAULT가 자동 적용된다고 단정할 수 없습니다.
- NULL을 허용하지 않으려면 DEFAULT만으로는 부족하고 NOT NULL 제약조건을 함께 고려해야 합니다.
- SQLD에서도 DEFAULT, NOT NULL, CHECK, UNIQUE, PRIMARY KEY 같은 제약조건의 역할 구분을 자주 봅니다.
- DEFAULT는 입력 편의성과 일관된 초기값 설정에 도움을 주지만, 무결성 보장의 전부는 아닙니다.

- 배타적이면 한 슈퍼타입 인스턴스가 하나의 서브타입에만 속하고, 중복이면 여러 서브타입에 동시에 속할 수 있습니다.
- 또한 모든 슈퍼타입 인스턴스가 반드시 서브타입에 속해야 하는지, 일부만 속해도 되는지도 구분합니다.
- 시험에서는 슈퍼타입/서브타입을 단순 상속 느낌으로만 보지 말고, 데이터 중복 감소와 업무 규칙 표현 관점에서 이해하는 것이 중요합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: 슈퍼타입은 공통 속성, 서브타입은 유형별 고유 속성을 표현한다.
- SQLD 포인트: 배타/중복 서브타입, 전체/부분 특수화 여부를 ERD 해석 문제로 자주 묻는다.
- 주의: 모든 서브타입이 항상 배타적이거나 항상 중복인 것은 아니며, 업무 규칙에 따라 결정된다.

3. OX 문제
슈퍼타입/서브타입 모델링에서 하나의 슈퍼타입 인스턴스가 반드시 여러 서브타입에 동시에 중복 소속되어야 하므로, 배타/중복 구분은 필요 없다.
