오늘 주제: 네트워크 - MTU와 TCP MSS

1. 핵심 요약
- MTU는 Maximum Transmission Unit으로, 한 네트워크 링크에서 한 번에 보낼 수 있는 최대 전송 단위입니다.
- 이더넷에서는 보통 MTU가 1500바이트로 자주 설명됩니다.
- MSS는 Maximum Segment Size로, TCP 세그먼트 안에 들어가는 실제 데이터(payload)의 최대 크기입니다.
- 즉 MTU가 “전체 IP 패킷 크기 한계”에 가깝다면, MSS는 “TCP 데이터 크기 한계”입니다.
- 일반적으로 MSS = MTU - IP 헤더 - TCP 헤더로 계산합니다.
- 예를 들어 MTU 1500, IPv4 헤더 20, TCP 헤더 20이면 MSS는 1460바이트입니다.
- MSS를 너무 크게 잡으면 경로 중간의 MTU보다 커져 IP 단편화가 생기거나 전송 문제가 생길 수 있습니다.
- TCP는 연결 설정 과정에서 MSS 옵션을 교환해 상대에게 자신이 받을 수 있는 최대 세그먼트 크기를 알려줄 수 있습니다.
- 시험에서는 MTU와 MSS를 헷갈리게 내기 좋으니, MTU는 패킷 전체 기준, MSS는 TCP payload 기준으로 잡으면 됩니다.

2. 자주 나오는 포인트
- MTU: 링크에서 전송 가능한 최대 IP 패킷 크기.
- MSS: TCP 데이터 최대 크기이며 IP/TCP 헤더는 제외.
- 단편화 방지 관점에서 MTU, MSS, Path MTU Discovery 개념이 함께 연결됩니다.

3. OX 문제
MSS는 TCP 세그먼트에서 실제 데이터(payload)가 들어갈 수 있는 최대 크기를 뜻하며, 일반적으로 MTU에서 IP 헤더와 TCP 헤더 크기를 제외해 계산한다.

오늘 주제: 운영체제 - DMA와 인터럽트 기반 입출력

1. 핵심 요약
- 입출력 방식은 CPU가 장치와 데이터를 어떻게 주고받는지에 대한 운영체제 핵심 개념입니다.
- 프로그램 입출력은 CPU가 장치 상태를 계속 확인하며 직접 데이터를 옮기는 방식이라 단순하지만 CPU 낭비가 큽니다.
- 인터럽트 기반 입출력은 장치가 작업 완료나 상태 변화를 CPU에 알려 주는 방식입니다.
- CPU는 계속 기다리지 않고 다른 작업을 하다가, 인터럽트가 오면 처리 루틴으로 전환합니다.
- DMA는 Direct Memory Access의 약자로, 장치와 메모리 사이의 데이터 블록 전송을 DMA 컨트롤러가 맡는 방식입니다.
- CPU는 전송 시작에 필요한 주소, 크기, 방향만 설정하고 실제 대량 전송에는 덜 관여합니다.
- 전송이 끝나면 DMA 컨트롤러가 인터럽트로 CPU에 완료를 알릴 수 있습니다.
- 그래서 디스크, 네트워크, 사운드처럼 대용량 데이터 입출력에서 CPU 부담과 오버헤드를 줄이는 데 유리합니다.

2. 자주 나오는 포인트
- 프로그램 입출력: CPU가 직접 계속 확인 → CPU 낭비 가능
- 인터럽트 입출력: 장치가 완료/상태 변화를 CPU에 알림

- 연결 설정은 3-way handshake, 연결 종료는 보통 FIN/ACK 기반 4-way handshake
