오늘 주제: SQL/DB - NATURAL JOIN과 조인 조건

1. 핵심 요약
- NATURAL JOIN은 두 테이블에서 이름이 같은 컬럼을 자동으로 찾아 조인 조건으로 사용하는 방식입니다.
- 예를 들어 사원 테이블과 부서 테이블에 둘 다 deptno 컬럼이 있으면, NATURAL JOIN은 deptno가 같은 행끼리 연결합니다.
- 동작 원리는 편해 보이지만, “같은 이름의 모든 컬럼”이 조건에 들어간다는 점이 중요합니다.
- 의도한 공통 컬럼이 deptno 하나였는데, 두 테이블에 name 같은 동일 이름 컬럼이 추가로 있으면 그것까지 조건에 포함될 수 있습니다.
- 그래서 결과 행이 예상보다 줄거나, 조인 의미가 달라질 수 있습니다.
- SQLD에서도 JOIN 문법은 자주 나오며, NATURAL JOIN은 ON/USING과 비교해서 이해하는 것이 좋습니다.
- ON은 조인 조건을 직접 작성하고, USING은 같은 이름의 특정 컬럼을 명시합니다.
- 시험에서는 NATURAL JOIN이 “동일 이름 컬럼 자동 매칭”이라는 점과, 명시적 조건이 없어도 조인이 수행된다는 점을 묻습니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: INNER JOIN은 조건 만족 행만, OUTER JOIN은 한쪽 테이블 행 보존, CROSS JOIN은 모든 조합입니다.
- NATURAL JOIN은 같은 이름의 컬럼을 자동 조건으로 사용하므로, 조인 기준 컬럼을 눈으로 확인하기 어렵다는 단점이 있습니다.
- ON은 자유롭게 조건 작성, USING은 동일 이름 컬럼 중 사용할 컬럼 지정, NATURAL JOIN은 동일 이름 컬럼 자동 사용입니다.

NATURAL JOIN은 두 테이블에서 이름이 같은 모든 컬럼을 자동으로 조인 조건에 사용하므로, 의도하지 않은 동일 이름 컬럼이 있으면 결과가 달라질 수 있다.

오늘 주제: 운영체제 - 페이지 교체 알고리즘 LRU와 OPT

1. 핵심 요약
- 페이지 교체 알고리즘은 메모리에 필요한 페이지가 없을 때 어떤 페이지를 내보낼지 정하는 방법입니다.
- FIFO는 가장 먼저 메모리에 들어온 페이지를 교체합니다.
- LRU는 “가장 오랫동안 사용되지 않은 페이지”를 교체합니다.
- 즉 LRU는 과거 사용 이력을 보고, 최근에 안 쓴 페이지는 앞으로도 덜 쓸 가능성이 있다고 가정합니다.
- OPT는 앞으로 가장 오랫동안 사용되지 않을 페이지를 교체하는 이론적 최적 알고리즘입니다.
- 하지만 OPT는 미래의 참조 순서를 알아야 하므로 실제 구현보다는 성능 비교 기준으로 쓰입니다.
- 시험에서는 FIFO, LRU, LFU, OPT의 기준을 서로 바꿔 내는 문제가 자주 나옵니다.

2. 자주 나오는 포인트
- FIFO: 적재된 지 가장 오래된 페이지 교체
- LRU: 최근 사용 시점이 가장 오래된 페이지 교체
- OPT: 앞으로 가장 늦게 사용될 페이지 교체, 이론적 최적

3. OX 문제
LRU 페이지 교체 알고리즘은 앞으로 가장 오랫동안 사용되지 않을 페이지를 예측해 교체하는 최적 교체 알고리즘이다.

오늘 주제: 정보보안 - CSRF 공격과 방어

1. 핵심 요약
- CSRF는 Cross-Site Request Forgery, 즉 사이트 간 요청 위조 공격입니다.
- 사용자가 어떤 서비스에 로그인된 상태라는 점을 악용합니다.
- 공격자는 사용자가 의도하지 않은 요청을 브라우저가 보내게 만듭니다.
- 서버는 요청에 정상 세션 쿠키가 포함되어 있으면 진짜 사용자 요청처럼 오해할 수 있습니다.
- 예를 들어 로그인된 상태에서 공격 링크를 눌렀더니 비밀번호 변경, 송금, 게시글 작성 요청이 전송되는 식입니다.
- SQL Injection은 SQL 질의문을 조작하는 공격이고, CSRF는 인증된 사용자의 요청을 위조하는 공격입니다.
- XSS는 브라우저에서 악성 스크립트를 실행시키는 공격이므로 CSRF와도 구분해야 합니다.
- 방어에는 CSRF 토큰, SameSite 쿠키, 중요 요청 재인증, Origin/Referer 검증 등이 사용됩니다.

2. 자주 나오는 포인트
- CSRF = 인증된 사용자의 권한을 악용한 요청 위조
- SQL Injection = DB 질의 조작, XSS = 악성 스크립트 실행, CSRF = 요청 자체를 위조
- 방어 키워드: CSRF Token, SameSite, 재인증, Origin/Referer 검증

3. OX 문제
CSRF는 사용자가 이미 인증된 상태를 악용해 원치 않는 요청을 보내게 만드는 공격이며, 단순히 서버 데이터베이스 질의문을 조작하는 SQL Injection과는 구분된다.

- `IN`은 값이 목록 중 하나와 같은지 확인하는 조건입니다.
- 반대로 `NOT IN`은 목록의 모든 값과 다르다고 판단되어야 TRUE가 됩니다.
- 그런데 `NOT IN` 목록이나 서브쿼리 결과에 NULL이 섞이면 “NULL과 다른가?”를 확정할 수 없어 UNKNOWN이 생깁니다.
- WHERE절은 TRUE인 행만 통과시키므로 UNKNOWN은 결과에서 제외됩니다.
- 그래서 `NOT IN (서브쿼리)`를 쓸 때 서브쿼리 결과에 NULL이 포함되면 예상보다 결과가 안 나올 수 있습니다.
- SQLD에서도 자주 보는 포인트는 NULL 비교는 `= NULL`이 아니라 `IS NULL`, `IS NOT NULL`을 써야 한다는 점입니다.
- 실무적으로는 `NOT EXISTS`를 쓰거나, 서브쿼리에서 `WHERE 컬럼 IS NOT NULL`로 NULL을 제거하는 방식도 자주 고려합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: NULL 비교 결과는 UNKNOWN이 될 수 있고, WHERE절은 TRUE만 통과시킵니다.
- SQLD 포인트: `NOT IN` 서브쿼리 결과에 NULL이 포함되면 결과가 예상과 달라질 수 있습니다.

오늘 주제: SQL/DB - 트랜잭션 격리 수준과 Non-repeatable Read

1. 핵심 요약
- 트랜잭션 격리 수준은 여러 트랜잭션이 동시에 실행될 때 서로의 작업을 어느 정도까지 보이게 할지 정하는 기준입니다.
- Dirty Read는 다른 트랜잭션이 아직 COMMIT하지 않은 데이터를 읽는 현상입니다.
- READ COMMITTED는 커밋된 데이터만 읽게 하므로 Dirty Read를 막을 수 있습니다.
- 하지만 같은 트랜잭션 안에서 같은 행을 두 번 읽는 사이, 다른 트랜잭션이 그 행을 수정하고 COMMIT하면 두 읽기 결과가 달라질 수 있습니다.
- 이것을 Non-repeatable Read, 즉 반복 불가능 읽기라고 합니다.
- 더 높은 격리 수준인 REPEATABLE READ는 같은 행을 반복 조회할 때 값이 바뀌는 문제를 줄이는 데 초점이 있습니다.
- SERIALIZABLE은 가장 강한 격리 수준으로 동시 실행 결과가 순차 실행처럼 보이게 하지만, 동시성은 낮아질 수 있습니다.
- SQLD에서도 Dirty Read, Non-repeatable Read, Phantom Read 차이를 자주 묻습니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: Dirty Read = 미커밋 데이터 읽기, Non-repeatable Read = 같은 행 재조회 값 변경.
- READ COMMITTED는 Dirty Read는 방지하지만 Non-repeatable Read는 발생 가능.
- 격리 수준이 높아질수록 일관성은 강해지지만, 일반적으로 동시성은 낮아질 수 있음.
