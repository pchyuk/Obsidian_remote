1. 핵심 요약
- TLS는 인터넷 통신에서 도청·변조·위조를 줄이기 위해 사용하는 보안 프로토콜입니다.
- HTTPS는 HTTP 메시지를 TLS로 보호해 전송하는 방식이라고 보면 됩니다.
- TLS 핸드셰이크는 본격적으로 데이터를 주고받기 전에 “누구와 어떤 방식으로 안전하게 통신할지” 정하는 절차입니다.
- 이 과정에서 서버는 인증서를 제시하고, 클라이언트는 인증서가 신뢰할 수 있는 CA에 의해 발급되었는지 확인합니다.

- 하지만 실제 대량 데이터 전송을 계속 공개키 암호화로 처리하면 너무 느립니다.
- 그래서 핸드셰이크로 안전하게 세션 키를 만든 뒤, 실제 데이터는 빠른 대칭키 암호화로 보호하는 구조가 일반적입니다.
- 즉 “공개키로 전부 암호화한다”가 아니라 “공개키 기반 절차로 세션 키를 안전하게 만들고, 이후 대칭키로 통신한다”가 핵심입니다.

2. 자주 나오는 포인트
- HTTPS = HTTP + TLS, 기본 포트는 443입니다.

- 시험에서는 공개키 암호화와 대칭키 암호화의 역할 분담을 자주 묻습니다.

오늘 주제: SQL/DB - JOIN 조건 USING과 ON 차이

1. 핵심 요약
- JOIN은 두 테이블의 행을 어떤 조건으로 연결할지 정하는 SQL 핵심 문법입니다.
- ON 절은 가장 명시적인 조인 조건 작성 방식입니다.
- 예를 들어 ON e.dept_id = d.id처럼 두 테이블의 컬럼명이 달라도 조인할 수 있습니다.
- 여러 조건을 AND로 추가하거나, 컬럼명이 다른 경우에도 유연하게 사용할 수 있습니다.
- USING 절은 두 테이블에 같은 이름의 컬럼이 있을 때 USING(컬럼명)처럼 간단히 씁니다.
- USING을 쓰면 조인 기준 컬럼이 결과에서 중복 표시되지 않고 하나로 합쳐져 보일 수 있습니다.
- NATURAL JOIN은 같은 이름의 컬럼을 자동으로 모두 조인 조건에 쓰므로 의도하지 않은 컬럼까지 조건이 될 수 있습니다.
- SQLD에서도 ON, USING, NATURAL JOIN의 차이는 조인 결과 해석 문제로 자주 나옵니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: ON은 컬럼명이 달라도 사용 가능하고 조건을 명시적으로 작성한다.
- SQLD 포인트: USING은 양쪽 테이블에 같은 이름의 컬럼이 있을 때 사용한다.
- 주의: NATURAL JOIN은 같은 이름 컬럼을 자동 사용하므로 예측하지 못한 결과가 나올 수 있다.

3. OX 문제
SQL의 USING 절은 두 테이블에서 조인에 사용할 컬럼명이 같을 때 해당 컬럼을 한 번만 지정해 조인할 수 있지만, 컬럼명이 다르거나 복잡한 조건이 필요하면 ON 절을 사용해야 한다.

오늘 주제: SQL/DB - 윈도우 함수 ORDER BY와 프레임

1. 핵심 요약
- 윈도우 함수는 행을 합쳐 줄이지 않고, 각 행을 유지한 채 분석 값을 계산하는 SQL 기능입니다.
- PARTITION BY는 계산 대상을 부서별, 고객별처럼 나누는 기준입니다.
- ORDER BY는 파티션 안에서 어떤 순서로 계산할지 정합니다.
- 누적합, 순위, 이전/다음 행 비교처럼 순서가 중요한 계산은 ORDER BY가 핵심입니다.
- 윈도우 프레임은 현재 행을 기준으로 어디부터 어디까지를 계산 범위에 넣을지 정합니다.
- 예: ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW는 처음 행부터 현재 행까지를 의미합니다.
- SQLD에서도 GROUP BY와 윈도우 함수 차이를 자주 묻습니다.
- GROUP BY는 여러 행을 하나로 줄일 수 있지만, 윈도우 함수는 원래 행 수를 유지합니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: PARTITION BY=계산 그룹 분리, ORDER BY=그룹 내부 순서 지정.
- SQLD 포인트: ROW_NUMBER, RANK, SUM() OVER 같은 함수는 OVER 절 구조를 해석할 줄 알아야 합니다.
- 주의: PARTITION BY만으로 누적합 순서가 보장되는 것은 아닙니다.

3. OX 문제
윈도우 함수에서 PARTITION BY만 지정하면 파티션 내부의 누적합 계산 순서가 항상 보장되므로 ORDER BY나 윈도우 프레임을 따로 고려할 필요가 없다.

오늘 주제: 네트워크 - CIDR과 서브넷 호스트 수 계산

1. 핵심 요약
- CIDR은 IP 주소 뒤에 /24, /27처럼 prefix 길이를 붙여 네트워크 범위를 표현하는 방식입니다.
- /27은 IPv4 32비트 중 앞의 27비트가 네트워크 부분이라는 뜻입니다.
- 남은 호스트 비트는 32 - 27 = 5비트입니다.
- 호스트 비트 5개로 만들 수 있는 전체 주소 수는 2⁵ = 32개입니다.
- 일반적인 IPv4 서브넷에서는 호스트 비트가 모두 0인 주소는 네트워크 주소로 사용합니다.
- 호스트 비트가 모두 1인 주소는 브로드캐스트 주소로 사용합니다.
- 그래서 실제 호스트에 할당 가능한 주소 수는 32 - 2 = 30개입니다.
- prefix 숫자가 커질수록 네트워크 부분은 길어지고, 호스트 수는 줄어듭니다.
- 반대로 /24보다 /27은 더 작은 서브넷으로 쪼갠 것이며, 한 서브넷에 들어갈 수 있는 장비 수가 줄어듭니다.

2. 자주 나오는 포인트
- 사용 가능 호스트 수: 2^(호스트 비트 수) - 2
- /27이면 호스트 비트 5개 → 전체 32개 → 사용 가능 30개
- 네트워크 주소와 브로드캐스트 주소는 일반 호스트에 할당하지 않는다는 점을 자주 묻습니다.

오늘 주제: SQL/DB - DML INSERT INTO SELECT

1. 핵심 요약
- INSERT는 테이블에 새로운 행을 추가하는 DML입니다.
- 기본형은 INSERT INTO 테이블명 VALUES (...)처럼 값을 직접 넣는 방식입니다.
- INSERT INTO ... SELECT는 SELECT 결과를 이용해 다른 테이블에 행을 삽입하는 방식입니다.
- 즉, 한 행씩 직접 입력하는 것이 아니라 조회 결과로 나온 여러 행을 한 번에 넣을 수 있습니다.
- 예를 들어 백업 테이블 생성 후 조건에 맞는 데이터만 복사할 때 자주 쓰입니다.
- 동작 원리는 SELECT가 먼저 결과 집합을 만들고, 그 결과의 각 행이 INSERT 대상 테이블에 들어가는 구조입니다.
- 대상 컬럼 목록과 SELECT 결과 컬럼의 개수·순서·데이터 타입이 맞아야 합니다.
- SQLD에서도 DML 기본 문법, INSERT 방식 차이, 제약조건 위반 여부를 자주 확인합니다.
- PRIMARY KEY, NOT NULL, CHECK, FOREIGN KEY 조건을 어기면 INSERT가 실패할 수 있습니다.

2. 자주 나오는 포인트
- 정처기/SQLD 공통: INSERT는 SELECT·UPDATE·DELETE와 함께 DML에 속합니다.
- INSERT INTO ... VALUES는 직접 값 삽입, INSERT INTO ... SELECT는 조회 결과 삽입입니다.
- 대상 컬럼과 SELECT 결과 컬럼의 개수·타입·제약조건 만족 여부를 확인해야 합니다.

3. OX 문제
INSERT INTO ... SELECT 문은 기존 테이블의 조회 결과를 이용해 다른 테이블에 여러 행을 한 번에 삽입할 수 있는 DML이다.
